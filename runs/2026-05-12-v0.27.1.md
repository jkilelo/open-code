# Run log -- open-code v0.27.1 -- 2026-05-12

**Persona:** Jeff.
**Trigger:** User asked: "can we borrow newer design patterns from
claude code and other terminals? e.g. fixed terminal sections in
multiple areas each providing static or dynamic feedback to users."

Honest answer: yes, with a caveat. The pattern that pays off is a
sticky bottom status panel during a turn (Claude Code / vim
statuslines / tmux). Full multi-pane TUI (Textual-style) would be
overkill for a CLI agent and break the "feels like a shell" model.

## Feature shipped

### LiveStatusPanel: sticky bottom panel during a turn

New classes in `ui.py`:

- `LiveStatusPanel` -- Rich Live-based status panel anchored to the
  bottom of the terminal. Auto-refreshes 4x/sec showing iter, model,
  tokens, wall time, tool counts, and the current action with a
  dots spinner.
- `NoOpPanel` -- drop-in replacement returned in plain/json/non-TTY
  modes so callers use the same try/finally without branching.

What the user sees in rich+TTY mode:

```
[ ... model text + tool calls scroll naturally above ... ]
+-- open-code ----------------------------------------------+
| [iter 2/25]  gemini-3.1-flash-lite-preview                |
| 3.1K in  16 out  2.1s  1 tools                            |
| (spinner) now: read_file(README.md)                       |
+-----------------------------------------------------------+
```

The panel is `transient=True`, so it clears cleanly when the turn
ends and is replaced by the existing one-line turn summary
(`[iters=N in=Nt out=Nt wall=Ns tools=N errs=N]`).

### Wiring

In `run_loop`:

- `panel = ui.live_panel(model, max_iters, session_id)` at the top
  of the function. Calls `panel.start()` once.
- At the start of each iteration: `panel.update(iter, in_tok,
  out_tok, tool_calls, tool_errors); panel.set_action("thinking...")`.
- In the `finally:` block: `panel.stop()` BEFORE the turn-end
  checkpoint + JSON event emission so the summary doesn't repaint
  over the panel.

In `cli.py`:

- New `--no-panel` flag. Sets `OPEN_CODE_NO_PANEL=1` in env so the
  UI's lazy panel-create check picks it up.

### Mode detection

`ui.live_panel(...)` returns:

- `NoOpPanel` if mode != rich (plain/json)
- `NoOpPanel` if `OPEN_CODE_NO_PANEL=1`
- `NoOpPanel` if `quiet=True`
- `NoOpPanel` if `console.is_terminal` is False (non-TTY stderr)
- `LiveStatusPanel` otherwise

All NoOpPanel methods are no-ops so the wiring is identical in
all modes.

## Honest trade-offs

- **Panel takes the bottom area of the terminal.** Rich's Live
  handles the redraw so tool calls + model text scroll naturally
  above. But if a tool produces a LOT of output, the panel and the
  output can briefly fight for repaints. Mitigation: small output
  (line-by-line) is smooth; multi-thousand-line output may flicker.
- **`transient=True` means the panel disappears on stop**, leaving
  no trace of what just happened. The `ui.turn_summary` line takes
  its place. This is intentional -- you can scroll up to see tool
  calls but the panel itself is ephemeral.
- **One panel per turn.** REPL-mode chained turns each get their
  own panel. Restarting the panel between turns means a brief
  flash but each turn's state is self-contained.
- **No multi-pane layout.** Two-pane (output + task list) was
  considered and rejected -- it would require capturing stdout
  redirects, which fights with our `_emit_json` path. Sticky footer
  is the right scope.
- **Auto-disabled in plain/json/non-TTY.** Users piping output get
  the existing scrolling-line UX, unchanged from v0.27.0. The
  fancy version is opt-in via "you're on a real terminal."
- **ASCII source enforcement caught two `.` chars** I typed at
  edit time. The probe_ascii_only guard locked the repo before
  the typo could escape into a commit. Working as designed.
