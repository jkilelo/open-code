# Run log -- open-code v0.28.0 -- 2026-05-12

**Persona:** Jeff.
**Trigger:** User asked: why does every open-code run emit these
three lines of stderr noise?

```
C:\...\pydantic\plugin\_schema_validator.py:40: UserWarning:
ImportError while loading the `logfire-plugin` Pydantic plugin,
this plugin will not be installed.

ImportError("cannot import name 'LogData' from
'opentelemetry.sdk._logs' ...")
```

## Root cause

Not open-code's bug. It's a dependency-version mismatch in the
user's global Python environment:

1. The user has `logfire` installed in
   `~/AppData/Roaming/Python/Python313/site-packages/`.
2. `logfire` registers a Pydantic v2 plugin via entry-point group
   `pydantic.plugins` (name: `logfire-plugin`).
3. When `google-genai` (which open-code depends on) imports
   `pydantic`, Pydantic discovers all registered plugins.
4. `logfire-plugin`'s setup tries
   `from opentelemetry.sdk._logs import LogData`.
5. The user's `opentelemetry-sdk` version no longer exports
   `LogData` (renamed/removed in a recent OTel release).
6. Pydantic catches the `ImportError`, emits a `UserWarning`, and
   skips the plugin. open-code runs fine but the warning leaks
   through to stderr on every startup.

This is environmental noise, not a behavioral bug. But it's three
lines of unfamiliar tracebacks at the top of every run -- the
opposite of "modern UX." Worth suppressing in open-code itself so
users on similar envs don't worry.

## Fix

Added a 17-line preamble at the top of `open_code.py`, BEFORE any
other imports (so it runs before pydantic's plugin scan fires):

```python
# Quiet third-party plugin-discovery noise before importing anything
# that triggers it. open-code itself doesn't use Pydantic plugins;
# the warnings come from other packages registered in the user's
# global site-packages (typically `logfire-plugin` which depends on
# a specific opentelemetry-sdk version). Must run BEFORE any import
# that loads google-genai / pydantic (which scans for plugins).
import os
os.environ.setdefault("PYDANTIC_DISABLE_PLUGINS", "1")
import warnings
warnings.filterwarnings(
    "ignore",
    message=r".*logfire-plugin.*",
    category=UserWarning,
)
warnings.filterwarnings(
    "ignore",
    message=r".*Pydantic plugin.*",
    category=UserWarning,
)
```

Why both env var AND warnings filter?

- `PYDANTIC_DISABLE_PLUGINS=1` -- Pydantic respects this and skips
  plugin discovery entirely. Belt.
- `warnings.filterwarnings(...)` -- catches the message even if a
  future Pydantic version ignores the env var or scans plugins
  before our preamble runs. Suspenders.

`os.environ.setdefault(...)` (not `os.environ[...] = ...`) so a
user who actively WANTS Pydantic plugins (e.g. running open-code
inside a logfire-instrumented project) can override:
`PYDANTIC_DISABLE_PLUGINS=0 python open_code.py ...`.

## Probes

No new probe. The fix is a stderr-noise suppression at module load;
adding a probe that checks "no UserWarning emitted on import" would
require subprocess + stderr capture, which is fragile. The visual
verification above is the test.

Full regression: 48/48 PASS. probe_ascii_only PASS.

## Honest trade-offs

- **Suppression hides genuine pydantic plugin breakage too.** If a
  legitimately useful plugin (e.g. a future security-validating one)
  fails to load, the user won't see why. Mitigation: the warnings
  filter is narrowly scoped to messages containing "logfire-plugin"
  OR "Pydantic plugin" -- specific enough to skip the noise, broad
  enough to catch the class. Other warnings still emit.
- **`PYDANTIC_DISABLE_PLUGINS=1` disables ALL plugin discovery.**
  open-code itself doesn't use pydantic plugins so this is safe.
  If a future open-code feature wanted pydantic plugin support,
  remove the env var line and rely on the message filter alone.
- **`setdefault` lets users opt in.** A user who actively wants
  pydantic plugins can run with `PYDANTIC_DISABLE_PLUGINS=0`.
