## #7 Repo-map (Aider-style symbol skeleton, Python-only)

PROMPT-PACK called for tree-sitter + networkx (~30-language coverage).
**Pragmatic v0.13 scope**: Python-only, stdlib `ast`, hand-rolled
PageRank. **Zero new deps.** Tree-sitter polyglot extension is v0.14+.

What it does:
- On every invocation (one-shot OR REPL session start), build a
  symbol skeleton of the repo and inject it into the system
  instruction as a `<repo-map>...</repo-map>` block.
- Each file gets a header (`# path/to/file.py`) + its top-level
  function / class signatures (no bodies).
- Cap: 4000 chars by default (~1000 model tokens).

Algorithm (see `repomap.py`):
1. `discover_files(cwd)` — `git ls-files *.py` (fallback: rglob),
   skip vendored dirs (.venv, node_modules, __pycache__, etc.)
2. `parse_file(p)` — stdlib `ast.parse`. Extract:
   - top-level functions / classes / methods (1 level deep)
   - all referenced names (Name, Attribute, Import targets)
3. `build_graph(symbols)` — directed edges: `A -> B` iff A
   references a name B defines.
4. `pagerank(edges, personalization=...)` — power method (20 iters,
   damping 0.85). Personalization: any file whose stem appears in
   the task text gets a boost.
5. `render_repomap(by_path, ranked, max_chars=4000)` — emit top
   files' signatures until the cap; append a truncation marker if
   more remain.

PageRank includes dangling-node mass redistribution (a common bug
in naïve power-method implementations — without it, scores leak to
zero on graphs with sink nodes). Probe test 4 asserts scores sum
to ~1.0.

## Honest trade-offs

- **Python-only.** Tree-sitter would extend to TypeScript, Go, Rust,
  C, Java, etc. — the cross-language version is v0.14+. For Jeff's
  persona (indie Python developer), this is high-value as shipped.
- **No file-content personalization.** Personalization currently
  matches the user's task text against file STEMS only. Searching
  inside file content would catch more refs but adds cost. Reach
  for tree-sitter v0.14 instead.
- **No cache.** Re-runs `discover_files + parse_file` every
  invocation. On the open-code repo (11 files), it takes ~50ms.
  If we ever target 1000+ files, add a
  `~/.open-code/cache/<encoded-cwd>/repomap.json` step keyed on
  file mtimes.
- **Method extraction is one level deep.** Class methods are
  rendered; methods inside nested classes are not. Most real
  code is one-level; nested classes are rare. Acceptable.
- **Function signatures use `ast.unparse`** for annotations and
  return types. Falls back to empty string on unparseable nodes
  (e.g., complex type expressions in old Python). No crashes.
