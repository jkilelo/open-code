# Run log -- open-code v0.34.0 -- 2026-05-14

**Persona:** Jeff.
**Trigger:** User runs Gemini behind a corporate wall that requires
Vertex AI mode with OAuth2 access tokens fetched from an internal
shell command (`helix auth access-token print -a`). Pasted the
production snippet and asked for parity through open-code's settings.

## What changed

`llm/_gemini.py`:

  - `GeminiClient.__init__` now dispatches on `extra["vertex"]["enabled"]`.
    Two paths:

      - **Developer API (default):** `genai.Client(api_key=api_key)`
        -- the unchanged path.
      - **Vertex AI (corporate):** `genai.Client(vertexai=True,
        project=..., location=..., credentials=..., http_options=...)`.

  - Module-level helpers `_resolve_vertex_credentials` and
    `_build_http_options` consume the `vertex.*` config.

  - New `_CommandCredentials` class. Dynamically subclasses
    `google.oauth2.credentials.Credentials` (lazy-imported) and
    overrides `refresh()` to re-run the user's shell command.
    Sets expiry to `now + 50min` so google-auth refreshes
    proactively before the 1-hour token life ends. Long REPL
    sessions don't 401 mid-task.

  - `api_key` is now `str | None`; the Vertex branch doesn't need
    one.

`llm/factory.py`:

  - The `GEMINI_API_KEY` check now skips when
    `extra.vertex.enabled` is truthy. The error message names the
    alternative so a misconfigured Vertex block surfaces a clear
    "set extra.vertex.enabled=true" hint rather than the developer-API
    text alone.

`tests/probe_gemini_vertex.py`:

  - 8 no-network checks that monkeypatch `genai.Client` and
    `subprocess.check_output`:

    1. `vertex.enabled` + `credentials_command` routes through
       `Client(vertexai=True, ..., http_options=HttpOptions(base_url=...))`.
    2. `_CommandCredentials.refresh()` re-runs the command and
       sets a 40-50 minute future expiry.
    3. Missing `project` -> `LLMConfigError`.
    4. `credentials_command` + `credentials_file` together ->
       `LLMConfigError` ("choose one, not both").
    5. Empty stdout from the command -> `LLMConfigError` ("empty").
    6. Factory skips `GEMINI_API_KEY` when `vertex.enabled`.
    7. `vertex` block is popped from `_extra` so it doesn't leak
       into per-call coercion.
    8. `vertex.enabled=false` still requires a `GEMINI_API_KEY`
       (no false-positive Vertex routing).

`README.md`:

  - New "Corporate Vertex AI (v0.34+)" section under Configuration.
    Side-by-side: the bare-SDK snippet the user pasted, and the
    `settings.json` equivalent. Documents the full `vertex.*` table,
    the credentials-precedence rules, the 50-min auto-refresh
    behavior, and the probe command.

## Settings shape

```json
{
  "llm": {
    "provider": "gemini",
    "model":    "gemini-3.1-flash-lite-preview",
    "extra": {
      "vertex": {
        "enabled": true,
        "project":  "my-gcp-project",
        "location": "global",
        "base_url": "https://corp-gateway.example.com",
        "credentials_command": "helix auth access-token print -a"
      }
    }
  }
}
```

Three credential strategies (pick one):

  - `credentials_command` -- shell command whose stdout is the
    access token. Re-run automatically on token expiry.
  - `credentials_file` -- path to a service-account JSON file.
    Standard google-auth refresh behavior.
  - Neither -- falls back to Application Default Credentials
    (`google.auth.default()`). Useful when
    `GOOGLE_APPLICATION_CREDENTIALS` or workload-identity is set.

## What didn't change

  - Public `make_llm_client(provider="gemini", ...)` signature.
    Vertex routing is purely additive through the existing
    `extra={}` channel.
  - The Anthropic / OpenAI / OpenAI-Chat adapters. Vertex is a
    Google-specific concern; other providers already supported
    `extra={"base_url": "..."}` for their own corporate-gateway
    cases.
  - Live smokes. `tests/_live_gemini_check.py` still hits the
    Developer API with an `AI...` key. A future
    `_live_gemini_vertex_check.py` would need real corporate
    credentials to exercise -- not something the public repo can
    pin to.

## Decision log

- **Why not a new `"vertex"` provider in `DEFAULT_MODELS`?** It's
  the same SDK and the same `GeminiClient` translation layer --
  only the `Client(...)` construction differs. A second top-level
  provider would duplicate `_gemini.py` minus six lines. The
  `extra.vertex` toggle keeps `provider="gemini"` honest:
  whichever wire you're on, the same neutral types flow through.

- **Why a dynamic subclass of `Credentials`?** Two reasons:

    1. Lazy import keeps `from llm import GeminiClient` from
       requiring `google-auth` for users who only use the
       Developer API.
    2. Overriding `refresh()` is the only clean way to make
       google-auth's transport ask for a new token before 401-ing
       on expiry. Without it, the user's 60-minute REPL session
       silently breaks at minute 60.

- **Why `shell=True` in the subprocess call?** Mirrors the user's
  production snippet exactly. Corporate auth tools rely on `PATH`
  resolution and shell features (env-var expansion, command
  chaining). The command name is sourced from the user's own
  `settings.json`, so no untrusted-input concern -- if an attacker
  can write to your project settings, they already control your
  agent.

- **Why default location `"us-central1"` and not `"global"`?**
  google-genai's documented default. User snippets often use
  `"global"` for endpoint routing but `"us-central1"` is what the
  SDK examples and Vertex docs assume. User can override.
