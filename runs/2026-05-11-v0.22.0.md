## Feature shipped

### #21 Skill prompt caching

Per-skill opt-in memoization of the expanded skill body. Skills with
`cache: true` in their frontmatter cache their expanded text (after
`$ARGUMENTS` substitution AND `` !`cmd` `` execution) for a TTL.

**Frontmatter format:**
```markdown
---
name: weekly-report
description: Builds the weekly review from git + tests
cache: true
---
This week's commits:
!`git log --since="7 days ago" --pretty=oneline`

Test summary:
!`pytest --collect-only -q | tail -5`
```

**Cache key:** `(skill_path, file_mtime, args_string)`.
- Editing the SKILL.md file bumps `mtime` -> cache miss -> re-expand.
- Different args -> different cache key -> fresh expansion.

**TTL:** 300s default; override via `OPEN_CODE_SKILL_CACHE_TTL` env.

**Bypass paths:**
- `expand_skill_body(skill, args, cwd, use_cache=False)` -- programmatic.
- REPL `/skill <name> --refresh [args]` -- `--refresh` token anywhere
  in the slash command bypasses cache for that one invocation.
- `skills.clear_skill_cache()` -- drops everything (used by tests).

**Why opt-in per skill:**
- Many skills include `!`date`` or `!`git status`` which are volatile
  by intent. Defaulting to cache=False preserves current behavior.
- Heavy-IO skills (like one that runs `pytest --collect-only` or
  `find` over a large tree) are obvious cache candidates and the
  author flags them explicitly.

**Wiring:**
- `skills.py`: new module-level `_EXPANSION_CACHE` dict; `_CACHE_TTL_SECS`
  read from env once at import.
- `Skill.cache: bool = False` (new field).
- `_parse_bool(fm.get("cache", ""))` in `load_skill_file`.
- `expand_skill_body(..., use_cache=True)` parameter.
- `clear_skill_cache()` public function for tests / explicit reset.
- REPL `/skill` parses `--refresh` and passes `use_cache=False`.

## Honest trade-offs

- **TTL is global, not per-skill.** A skill that needs a different
  cache lifetime (say, weekly) has to either set `OPEN_CODE_SKILL_CACHE_TTL`
  globally or use `cache: false` and accept re-expansion. Future
  polish: `cache: 3600` syntax for per-skill TTL.
- **Process-local cache.** `_EXPANSION_CACHE` is a module dict; it
  doesn't survive `open-code` exit. Each new invocation starts cold.
  Acceptable: most cache wins are inside long REPL sessions, where
  the same skill gets invoked repeatedly within minutes.
- **No max-size cap.** A long REPL session could accumulate thousands
  of cached entries if the user invokes lots of distinct skills with
  distinct args. Hasn't shown up in practice (typical sessions invoke
  3-5 skills); future polish: LRU eviction at N=500.
- **Cache hits ignore changes to FILES the `!`cmd`` reads.** A
  cached `!`cat README.md`` doesn't see edits to README.md until
  the TTL expires (or the user `/skill --refresh`s, or the SKILL.md
  itself is touched). This is the intended caching contract: it's
  faster but slightly stale. Documented above.
- **No invalidation hook.** A `PostToolUse` hook can't say "invalidate
  the skill cache after `write_file`". For now: trust TTL + manual
  `/skill --refresh`. If this becomes painful, add a hook payload
  shape that lets hooks call `skills.clear_skill_cache()`.
