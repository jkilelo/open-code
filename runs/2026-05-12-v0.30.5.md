## What got fixed

### 1. `requirements.txt` was stale + Gemini-locked

Pinned `google-genai>=1.75.0` while the README declared `>=2.1.0`
required. Anthropic and OpenAI installs were nowhere in the file
even though both providers are first-class. A user who ran
`pip install -r requirements.txt` then switched to Anthropic via
settings would hit `LLMConfigError: install anthropic` with no
breadcrumb pointing back to the doc.

Fix: reorganized into three sections -- core deps (always
required), default LLM provider (`google-genai>=2.1.0`), and
commented-out optional installs for the other two providers.

```
# Always required.
python-dotenv>=1.0.0
rich>=14.0.0
prompt_toolkit>=3.0.0

# Default LLM provider (Gemini). Switch via .open-code/settings.json
google-genai>=2.1.0

# Optional -- install only if you set the matching provider in settings.
# pip install "anthropic>=0.101.0"
# pip install "openai>=2.36.0"
```

### 2. README quickstart was POSIX-only

`export GEMINI_API_KEY=...` doesn't work on Windows PowerShell
(`$env:`) or cmd (`set`). The `.env` path -- which works
cross-platform via python-dotenv -- wasn't even shown in the README,
only in LEARN.md.

Fix: rewrote quickstart with `.env` as primary, three shell variants
(bash/PowerShell/cmd) shown for users who prefer environment
exports, and explicit switch-provider snippets for Anthropic +
OpenAI.

### 3. Gemini model name mismatch across the codebase

| File                           | Value                              |
|--------------------------------|------------------------------------|
| `open_code.DEFAULT_MODEL`      | `gemini-3.1-flash-lite-preview`    |
| `llm/factory.DEFAULT_MODELS["gemini"]` | `gemini-3.1-flash-lite`    |
| README provider table          | `gemini-3.1-flash-lite`            |
| README settings example        | `gemini-3.1-flash-lite-preview`    |

Both names happen to work (the non-preview is in
`MODEL_FALLBACK_CHAIN`) but a user comparing the docs scratches
their head.

Fix: aligned all four to `gemini-3.1-flash-lite-preview` (what
the CLI actually uses on a settings-less run, and what every
live session in this repo's `runs/` has logged).

### 4. `--model` / `OPEN_CODE_MODEL` undocumented

The v0.30.2-added resolution chain (`--model` > `OPEN_CODE_MODEL`
env > `settings.llm.model` > `settings.model` > `DEFAULT_MODEL`)
was internal-only. Use case "try one prompt against a stronger
model without editing settings" was undiscoverable.

Fix: new "Overriding the model for one invocation" subsection
under Configuration with the priority chain + two CLI examples.

### 5. `openai` vs `openai_chat` distinction was opaque

The provider table called them "modern" and "legacy / OSS-compat"
-- accurate but uninformative. A user picking a provider didn't
know that `openai` is the right default for native accounts and
`openai_chat` is for Groq/vLLM/Anthropic-OpenAI-shim/Azure-Chat
deployments.

Fix: side-by-side decision table covering when to pick each, plus
a `base_url` example showing how to point `openai_chat` at Groq:

```json
{"llm": {"provider": "openai_chat",
          "model":    "llama-3.3-70b-versatile",
          "extra":    {"base_url": "https://api.groq.com/openai/v1"}}}
```

### 6. Reasoning-token budget gotcha undocumented

`gpt-5-mini` allocates ~128 reasoning tokens before producing any
text. With `max_output_tokens=256` and the default reasoning
effort, structured-output calls come back empty because reasoning
consumed the budget. I rediscovered this during v0.30.1 testing;
users would rediscover it painfully without docs.

Fix: new "Gotchas worth knowing" subsection covering five
provider-specific traps:

  - gpt-5 reasoning eats budget invisibly
  - Anthropic `max_tokens` is mandatory + bounds thinking+text
  - `gpt-5.5-*` not yet GA on standard tier
  - Haiku 4.5 rejects adaptive thinking (adapter handles it)
  - Switching providers mid-session breaks signatures -- use `/clear`

### 7. INSTALL.md confusable for open-code install

INSTALL.md is for the **persona-mvp-kit** (parent kit, the
development-methodology bundle). A new user clicking around in the
repo might assume it's open-code's install guide and follow
irrelevant `chmod +x .claude/hooks/*.sh` instructions.

Fix: 4-line banner at the top: "Not the open-code install guide.
This document covers the persona-mvp-kit... If you want to install
and run the open-code CLI agent, see README.md."

### 8. `_live_*_check.py` smoke scripts invisible

The four re-runnable provider verification scripts
(`_live_gemini_check.py`, `_live_anthropic_check.py`,
`_live_openai_check.py`, `_live_openai_chat_check.py`) test 25
protocol surfaces between them. They're the fastest way for a
new user to confirm "my key + SDK + network all work" before
debugging a real session, but nothing in the user-facing docs
mentioned them.

Fix: "Verify your setup" block right under the quickstart pointing
at all four with one-line descriptions.

to date" -- where the honest answer was "mostly, with seven real

you change behavior, change the docs in the same commit" -- I

include a README touch in the same commit, not deferred.
