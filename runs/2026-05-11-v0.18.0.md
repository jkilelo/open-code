## Feature shipped

### #17 Sticky session permissions

The interactive `ask` prompt in REPL mode now offers four choices
instead of two:

```
  ? run_shell({"command": "git status"}) [y=once / s=session / a=always / n=no]:
```

- `y` → allow this one call (unchanged from before)
- `s` → allow this tool for the rest of THIS REPL session
  (in-memory; cleared by `/clear`)
- `a` → allow this tool always: persists a rule to
  `<cwd>/.open-code/settings.local.json` under
  `permissions.always_allow`
- `n` → deny

**The `always_allow` precedence tier.** A wrinkle showed up while
testing: a project's `settings.json` might declare
`permissions.ask = ["run_shell"]`. If the user picks "always" in the
REPL, that writes `permissions.allow = ["run_shell"]` to
`settings.local.json`. The merged result then has BOTH ask and allow
for `run_shell` — and the old evaluation order (deny > ask > allow)
meant ask still won. The user would keep getting prompted forever.

Fix: a new precedence tier `always_allow` slots between deny and ask.
A rule in `permissions.always_allow` wins over a competing ask rule
in any higher layer, while still respecting a deny rule.

New rule evaluation order: **deny > always_allow > ask > allow > default-allow**.

**Wiring:**
- `settings.PermissionRules.always_allow: list[str]` — new field.
- `settings._merge` recognizes the key in deep-merge.
- `settings.load_layered_settings` reads `permissions.always_allow`.
- `settings.evaluate_permission` checks it after deny, before ask.
- `open_code._persist_sticky_rule(cwd, tool)` — writes to
  `permissions.always_allow` in `.open-code/settings.local.json`;
  preserves other keys; idempotent; raises `RuntimeError` rather
  than silently clobbering malformed JSON.
- `run_loop`'s ask-prompt: parses `s` / `a` and either sets
  `settings._sticky_allow.add(tool)` (in-memory bypass for this
  REPL session) or calls `_persist_sticky_rule(...)`.
- Permission-gate fast-path: a tool in `settings._sticky_allow`
  short-circuits ask → allow without re-prompting.

## Honest trade-offs

- **Sticky is tool-name granular, not (tool, args) granular.** Picking
  "always" for `run_shell` will skip the prompt for `run_shell` calls
  with ANY arguments. If a user wants finer scope (e.g., "allow
  `run_shell(git *)` but not `run_shell(*)`"), they have to hand-edit
  `settings.local.json`. We deliberately chose tool-name simplicity for
  the prompt UX — too many decisions per prompt = decision fatigue.
- **`always` lands in `settings.local.json`, not the committed
  `settings.json`.** This was the safer default: a teammate cloning
  the repo doesn't automatically get the prior user's "always" grants.
  If a team WANTS to share the rule, they move it to `settings.json`.
- **The new `always_allow` tier is a real semantic change** to the
  rule evaluation function — even though it only activates when the
  field is populated, anyone whose mental model was "allow always
  loses to ask" needs to update. Documented in
  `settings.evaluate_permission` docstring.
- **In-memory `_sticky_allow` is a private attr on Settings.** It's
  not part of the dataclass schema; it's lazily set by run_loop on
  user input. Reasonable: it's per-REPL-session ephemeral state, not
  config. If we ever serialize Settings (we don't), this attr is
  invisible to JSON dumps. Trade-off vs. proper dataclass field:
  cleaner serialization shape.
- **No `/permissions clear` REPL command yet** to revoke a sticky-session
  grant. The user can `/clear` to start a fresh session (everything
  gone) or hand-edit `settings.local.json`. Future polish.
