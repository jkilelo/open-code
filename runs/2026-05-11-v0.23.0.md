## Feature shipped

### #22 Plugin system (without marketplace)

Plugins are bundles of skills + agents + output styles installed
under conventional directories. v0.23.0 ships the discovery +
aggregation half; a "marketplace / registry" is deliberately out of
scope (git clone is fine, and a curated index would require trust
infrastructure we don't have).

**Directory layout:**
```
<cwd>/.open-code/plugins/<plugin-name>/    (project-installed)
~/.open-code/plugins/<plugin-name>/        (user-installed)

each plugin/:
  plugin.json
  skills/<skill-name>/SKILL.md
  agents/<agent-name>.md  (or agents/<name>/AGENT.md)
  output-styles/<style-name>.md
```

**plugin.json schema:**
```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "...",
  "homepage": "https://...",
  "exposes": {
    "skills":        ["greet"],
    "agents":        ["counter"],
    "output_styles": ["zen"]
  }
}
```

**Precedence (lowest to highest):** built-in → plugin → user → project.
The user's project files always win. Same-name plugin from
`~/.open-code/plugins/` is overridden by same-name plugin from
`<cwd>/.open-code/plugins/`.

**New module: `plugins.py` (~155 LOC)** — public API:
- `Plugin` dataclass: name, version, description, source ("project"|"user"),
  path, and three exposes lists.
- `discover_plugins(cwd)` — project + user dirs, with project winning
  on name collision.
- `list_plugin_skill_dirs(cwd)` — for skills.py aggregation.
- `list_plugin_agent_files(cwd)` — for subagents.py aggregation (not
  wired in v0.23.0; documented).
- `list_plugin_output_styles(cwd)` — for output_styles.py.
- `render_plugin_listing(plugins)` — for `--list-plugins`.

**Aggregation wiring:**
- `skills.discover_skills(cwd)` now ALSO loads plugin-provided skills,
  then local skills override them on name collision.
- `output_styles.list_available(cwd)` reports plugin styles with source
  `plugin:<plugin-name>`.
- `output_styles.resolve_overlay(name, cwd)` checks plugins between
  user files and built-ins.

**NOT shipped in v0.23.0:**
- Plugin-provided hooks. Hooks need the v0.14.2 trust prompt; plugin
  install isn't tied into that flow yet. Adding it would require a
  per-plugin trust decision and a UX for it. Documented as carry.
- Plugin-provided agents discovery aggregation into `subagents.py`.
  The plumbing exists (`list_plugin_agent_files`) but I didn't wire
  subagents.discover_agents to call it — same trust concern (agents
  can run tools).
- Plugin install/uninstall commands. For now: `mkdir + cp` or
  `git clone https://.../my-plugin .open-code/plugins/my-plugin`.

## Honest trade-offs

- **No plugin marketplace.** A curated registry would require trust
  signing, vetting, an SDK, etc. Scoped out. Plugins install via
  `git clone` or copy; trust is implicit (it's in your CWD).
- **No plugin-provided hooks.** Plugin hooks would need to plug into
  the v0.14.2 trust prompt. Adding the prompt UI per-plugin is the
  right design but more surface than this commit's scope allows.
  Without trust, a plugin's hook would be RCE-on-install — exactly
  the bug the brutal review caught in v0.14.0. Stays excluded until
  we wire it carefully.
- **Plugin agents discovery aggregation is half-built.** The
  `list_plugin_agent_files` helper exists but `subagents.discover_agents`
  doesn't call it yet. Same trust concern as hooks (agents can run
  tools). Future commit.
- **No `--upgrade` / `--list-installed-versions` polish.** Users
  manually `git pull` inside the plugin dir. Acceptable v1.
- **Plugin manifest validation is permissive.** Unknown fields are
  ignored (forward-compat). Missing `name` falls back to the
  directory name. Malformed JSON is silent. This favors resilience
  over strictness; a `--strict-plugins` flag could be added later.
- **One layer between user and project for plugin styles.** The
  precedence is now builtin < plugin < user < project. That's four
  levels of indirection for output styles — discoverable but worth
  documenting prominently in `--list-styles` output.
