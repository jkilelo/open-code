## Feature shipped

### #11 Shadow-git checkpointing

A side git repository at `.open-code/checkpoints.git/` that snapshots
the workspace independently of the user's real `.git`. Lets the model
make changes and lets the user `/restore` to any prior point without
polluting their own commit history.

**New module: `checkpoints.py` (~217 LOC)** -- public API:
- `is_initialized(cwd) -> bool`
- `init_shadow_repo(cwd) -> (ok, msg)` -- idempotent
- `snapshot(cwd, label) -> (sha | None, msg)` -- `--allow-empty` so even
  no-change snapshots get a sha (markers for "what existed before X")
- `list_checkpoints(cwd, limit=20) -> [{sha, short_sha, ts, label}]`
- `resolve_ref(cwd, ref) -> sha | None` -- handles short shas + HEAD~N
- `diff_summary(cwd, from, to)` -- `git diff --stat` for preview before restore
- `restore(cwd, ref) -> (ok, msg)` -- `read-tree --reset -u` + `clean -fd`;
  exclude file keeps `.open-code/` and `.git/` safe.

**Design choices:**
- Bare git dir + `--work-tree=<cwd>` for every command, so we never
  pollute the user's real `.git` index. The shadow repo's worktree is
  the user's project, but its index/refs are entirely separate.
- `info/exclude` written at init-time with: `.open-code/`, `.git/`,
  `node_modules/`, `__pycache__/`, `*.pyc`, `.venv/`, `venv/`, `.tox/`,
  `.pytest_cache/`, `.mypy_cache/`, `.ruff_cache/`, `dist/`, `build/`,
  `*.egg-info/`. So a snapshot doesn't drag in caches or our own state.
- Local-only `user.email` + `user.name` config so commits don't fail
  when the user has no global git identity (or when they're behind
  a no-reply-email-only setup like the one referenced in our auto-memory).
- `commit.gpgsign=false` locally so signed-commit setups don't slow
  snapshots down.
- `snapshot()` is best-effort: missing git binary returns
  `(None, "git not available")` -- never crashes the agent loop.
- `restore()` does NOT prompt the user. The caller (REPL `/restore`)
  must confirm first; CLI/automation callers must opt in explicitly.

**Wiring:**
- `settings.Settings.auto_checkpoint: bool = False`. Set by:
  - `settings.json {"checkpoints": {"auto": true}}`, OR
  - `--auto-checkpoint` CLI flag.
- `sessions.SessionStore.append_checkpoint(session, sha, label, phase)`
  records `kind:"checkpoint"` events in the JSONL transcript with
  `short_sha`, `phase` in {"turn-start", "turn-end", "manual"}, and `ts`.
- `run_loop` in `open_code.py` takes a turn-start snapshot when
  `settings.auto_checkpoint` is True, just after the `SessionStart`
  hook. Failures are swallowed (snapshot is a luxury, not a guarantee).
- REPL slash commands in `repl.py`:
  - `/checkpoints` -- list recent shadow commits with timestamps
  - `/checkpoint [label]` -- manual snapshot now
  - `/restore <ref>` -- confirms (shows `git diff --stat` preview),
    takes a safety pre-restore snapshot, then restores.

- **Idempotent init**: second `init_shadow_repo` returns
  `ok=True, msg="already initialized"`.
- **Snapshot + list**: three sequential snapshots (one with content
  unchanged via `--allow-empty`, one with a real edit) produce three
  distinct shas; `list_checkpoints` returns newest-first.
- **Restore round-trip**: file content reverts to v1; a file added in
  v2 is removed by `git clean -fd`. The v2 commit is still resolvable
  by sha after the restore (objects don't go away -- HEAD just moved).
- **Exclude guard**: file inside `.open-code/secret.txt` is NOT in
  `ls-tree HEAD` after a snapshot -- the `info/exclude` worked.
- **Event semantics**: `append_checkpoint` writes a JSONL line with
  `kind:"checkpoint"`, full sha, short_sha (first 10), label, phase.
- **Graceful degrade**: monkeypatched `_git_available` to return False;
  `snapshot()` returns `(None, "git not available")` without raising.

## Honest trade-offs

- **Snapshot speed on big repos.** `git add -A` over a huge tree is
  not free. We default to OFF and let the user opt in via
  `--auto-checkpoint` or `settings.json`. The `info/exclude` knocks
  out the common slow paths (`node_modules/`, build dirs, virtualenvs).
- **Disk usage grows over time.** Each turn produces a commit, even
  empty ones (`--allow-empty`). Future improvement: `git gc --auto`
  call when shadow repo exceeds some size, or a `/checkpoints prune`
  REPL command. For now the user can `rm -rf .open-code/checkpoints.git`
  to reset -- they won't lose source code, only checkpoint history.
- **Restore is destructive.** We took two precautions: an explicit
  type-the-word-"restore" confirmation in the REPL, AND a safety
  pre-restore snapshot taken automatically so the user can roll
  forward. But: if the user's edits-in-progress aren't yet captured
  by ANY snapshot (e.g., they ran `--no-auto-checkpoint`), they lose
  unsaved work. Document loudly in /help.
- **Live REPL evidence is hand-tested.** The probe covers shadow-git
  module + event semantics. The slash-command flow (interactive
  confirmation) isn't easily probed without a tty harness. Future:
  hook the REPL into a pseudo-stdin probe.
- **Mode-vs-rule layer.** Checkpoints fire regardless of
  `settings.mode`. Even `plan` mode (which forbids `write_file`) will
  take a turn-start snapshot. That's a feature: the snapshot records
  state at task entry, useful even for read-only sessions.
