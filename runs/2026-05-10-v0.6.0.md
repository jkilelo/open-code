## Features shipped

### #2 Settings hierarchy + permission rules

New module `settings.py` (167 LOC). Three-tier merge:

1. `~/.open-code/settings.json` (user)
2. `<cwd>/.open-code/settings.json` (project, committed)
3. `<cwd>/.open-code/settings.local.json` (gitignored)

Schema (Claude Code-compatible subset):
```json
{
  "model": "gemini-3.1-flash-lite-preview",
  "max_iterations": 25,
  "permissions": {
    "allow": ["read_file(*)"],
    "ask":   ["write_file(*)"],
    "deny":  ["run_shell(rm -rf *)", "run_shell(sudo *)"]
  },
  "hooks":     { "disabled": false },
  "statusLine":{ "template": "...", "enabled": true }
}
```

Matcher syntax:
- `Tool` -- matches any args
- `Tool(spec)` -- fnmatch against JSON args OR any string arg value
- `Tool(/regex/)` -- re.search against JSON-serialized args

Evaluation order: **deny > ask > allow > default-allow**.

Evaluated BEFORE PreToolUse hooks so deny is deterministic regardless
of hook behavior.

ask in REPL -> y/n prompt; in one-shot -> auto-decline (with refusal
event logged).

`hooks.disabled: true` short-circuits PreToolUse/PostToolUse calls
in run_loop.

CLI flag precedence: explicit flags > env vars > settings.model /
settings.max_iterations. CLI defaults are only overridden by
settings when the user didn't pass them.

## Layered defense

open-code's security model is now three layers deep:

1. **OS-level path sandbox** -- `tool_write_file` refuses paths
   outside CWD (sticky, can't be hot-overridden by settings)
2. **Built-in destructive denylist** -- 30 patterns in `tools.py`,
   independent of settings (so a user can't accidentally allow
   `rm -rf /` via a permissive rule)
3. **Settings permission rules** -- user-customizable allow/ask/deny
4. **Hooks** -- last word on per-call gating (can rewrite args, log
   to audit trail, soft-block via Stop)

Hits in (1) or (2) ALWAYS block. Settings rules then layer on top.
Hooks layer on top of that. A permissive setting can't unlock the
hard-coded denylist; a denying hook can override an allowing setting.
