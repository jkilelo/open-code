## Three Claude-Code-style features shipped

### 1. Interactive REPL mode

`open_code` with no task argument drops into a persistent conversation.
Each prompt becomes one turn of the agent loop; the session persists
across turns in the same JSONL file; `Ctrl+D` (or `/exit`) leaves.

Slash commands (Claude-Code-style):

| Command | Action |
|---------|--------|
| `/help` | Show command list and @-file usage |
| `/exit`, `/quit` | Leave the REPL (also Ctrl+D) |
| `/clear` | Start a fresh session (forget context, new UUID) |
| `/sessions` | List recent sessions in this CWD |
| `/switch <uuid>` | Switch to a different session by UUID |
| `/cost` | Show cumulative cost (input/output tokens, fallbacks, refusals) |
| `/model <name>` | Switch model used for subsequent turns |
| `/dump` | Print the path of the JSONL transcript |

Ctrl+C cancels the in-flight turn but doesn't exit the REPL.
`readline` (when available, POSIX) gives line editing + history.

### 2. OPEN_CODE.md project context

Walks up from CWD looking for `OPEN_CODE.md`. First match wins.
Content is appended to the system instruction as a `## Project context`
block, capped at 50KB.

Tested live: a project with `OPEN_CODE.md` saying "Type hints required
on all public functions, one-line docstring" → the model wrote
`hello.py` with `-> None` return type and a docstring even though the
prompt didn't mention either. Project context genuinely shaped output.

### 3. @-file references in prompts

`summarize @README.md and @src/main.py` → before the model sees the
prompt, both files are read and prepended as `<file path="...">…</file>`
blocks. The literal `@path` token stays in the prompt so the model can
refer to the file naturally.

Smart parsing:
- URLs (`https://...`) not treated as file refs
- Missing paths left as literal text (no error, no injection)
- Trailing punctuation (`@README.md.`) stripped before resolving
- Same ref twice dedup'd (no double injection)
- 200KB cap per file with `[...truncated]` marker
- Subdirectory paths work (`@src/main.py`)

System instruction tells the model the `<file path="...">` blocks are
"already read for you — don't call read_file on them again." This
prevents the defensive re-read I observed in the first smoke test.

## Honest trade-offs

- **REPL stdin pipe rendering**: When tested with piped stdin, the
  `> ` prompt and model output interleave in stdout. In an interactive
  TTY this is fine because the prompt appears on its own line; piped
  it looks slightly scrambled. Common to most REPLs, not a bug.
- **REPL doesn't restream prior history**: switching sessions via
  `/switch` loads history into Gemini's context but doesn't echo it
  to the user. If Jeff wants to see what was in a session, he can
  `cat $(open_code ... /dump)` or use `--list-sessions` + the JSONL.
- **OPEN_CODE.md is unbounded by the agent's prompt size budget**:
  capped at 50KB. If Jeff writes a 200KB OPEN_CODE.md, it gets
  truncated with a `[...truncated]` marker. He'll notice.
- **@-file refs use any path that exists**: no path-sandbox enforcement
  on reads (only writes have sandbox). If Jeff types `@/etc/passwd`,
  it reads it. This is intentional — read access mirrors what
  read_file already does.
- **`tools.py` extraction was lossless**: every function/constant
  preserved, just relocated. Probes update one import line.

delivered, regression-clean, live-tested. The REPL with cross-turn
memory + slash commands matches Claude Code's primary UX shape; the
OPEN_CODE.md context concretely changes code the model writes; @-file
refs collapse the most common "first read this, then..." prompt
pattern into a single line.
