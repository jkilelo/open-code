## What changed

| Area | Before (v0.2.1) | After (v0.3.0) |
|------|-----------------|----------------|
| **Storage** | SQLite at `~/.open-code/sessions.db` with two tables and an auto-increment integer PK | File-per-session JSONL at `~/.open-code/projects/<encoded-cwd>/<uuid>.jsonl`. Append-only. Each line a JSON event. |
| **Session IDs** | INTEGER auto-increment | UUID4 (`f227d5ba-5906-40dc-8d85-0e46aae03f03`) |
| **Index** | SQLite indexes on `(cwd, last_active_at)` | The filesystem itself -- `projects/<encoded-cwd>/` is one directory per project |
| **Event kinds** | Only `messages` (user/model role) | `session` / `msg` / `metrics` / `fallback` / `refusal` / `end` |
| **`--resume`** | Most recent in CWD (still) | Same + `--resume-id <uuid>` for specific sessions |
| **Concurrency** | SQLite MVCC | One file per session -> no contention except on the same UUID |
| **Inspectability** | Binary blob, needs `.dump`/`sqlite3` CLI | `cat`/`grep`/`tail -f` work; one event per line |
| **Cost across `--resume`** | Per-invocation only | Cumulative: `[open-code:cumulative]` line re-reads metrics events |
| **Audit trail** | None | Tool refusals + model fallbacks logged as events |
| **Stream-error survivability** | Nothing on disk if stream crashed | Session header + user message + end event survive. Partial *model output* still doesn't (deferred). |
| **File layout** | Single 1062-line file | Split: `open_code.py` (970) + `sessions.py` (479) |

## Design patterns brought from Claude Code

1. **File-per-session JSONL** with append + flush per event. Mirrors how Claude Code stores transcripts in `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl`. Result: filesystem inspection is a first-class affordance; corruption is bounded to one session.

2. **Encoded CWD as a directory name.** `C:\Users\jeff\foo` -> `C-Users-jeff-foo`. The filesystem layout is self-documenting: when Jeff `ls ~/.open-code/projects/`, he sees his projects.

3. **UUID session IDs.** No coordination needed; no integer PK to mint. The UUID *is* the filename, the URL, and the resume key.

4. **Append-only event log with `kind` discriminator.** Not just messages -- `metrics`, `fallback`, and `refusal` events live next to messages chronologically. The session file becomes an audit log.

5. **`--resume` vs `--resume-id` distinction.** Claude Code's `--continue` (most recent) vs `--resume <id>` (pick specific) maps directly. Two clear UX paths for two different intents.

6. **Migration with rollback escape hatch.** Old `sessions.db` is renamed to `sessions.db.migrated` (not deleted). User can delete or restore.

7. **Aggregate metrics across resume chains.** Reading prior `metrics` events from disk lets `--show-metrics` report the lifetime cost of a session -- directly serving Jeff's "I switched for cost reasons" persona goal that v0.2 quietly missed.

### Stream-error survivability (improved, not fully closed)

`probe_stream_error.py` mocks a `ConnectionError` mid-stream. With JSONL + per-event flush:

```

  [session] task='test'
```

What's on disk: session header, user message, end event with exit code. **Better than v0.2.1** (which had only the user message). What's still missing: the *partial model text* that was already streamed to stdout but not yet committed as a `msg` event. Closing this fully requires saving stream chunks incrementally -- deferred to v0.4.

### Cumulative metrics across --resume -- closing the carried gap

The brutal review of v0.2.0 flagged `--show-metrics doesn't accumulate cost across --resume`. v0.3 closes this: every iter writes a `metrics` event; `--show-metrics` re-reads them and prints `[open-code:cumulative] iters=... input_tok=... output_tok=... fallbacks=... refusals=...`.

Tool refusals (path sandbox + denylist) are now logged as events:

## Honest trade-offs

- **Lost SQL's expressiveness.** I no longer have a SQL query layer for sessions. If Jeff ever wants `SELECT ... FROM messages WHERE ... GROUP BY model` that's now `cat *.jsonl | jq`. For this app that's a feature, not a bug -- but worth naming.
- **Lost atomic multi-statement transactions.** Not actually used in v0.2.1 (every message was its own commit), so no regression.
- **Filesystem is now load-bearing.** A corrupted directory or filesystem error affects open-code more directly than a SQLite blob. But it's also more recoverable: `head -1` the JSONL to find a session header; broken sessions just get skipped.
- **`--list-sessions` is a directory scan.** At our scale (dozens to hundreds of sessions per project) this is sub-100ms. At thousands of sessions per project it'd matter. Document and revisit if it ever does.
- **The `--resume`-by-default-most-recent uses mtime.** If `find_latest_for_cwd` lands on a file whose first line isn't a valid `session` header (e.g. corrupted), it falls through to the next candidate. Tested via the `_session_from_path` returning None path.
- **Partial stream content still doesn't survive crashes.** The honest claim: header + user msg + end event survive; the assistant's partial text does not. Closing this fully requires saving stream chunks incrementally. Deferred to v0.4.
