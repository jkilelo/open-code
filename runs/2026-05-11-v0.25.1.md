## Feature shipped

### `ui.UI.prompt()` -- prompt_toolkit input-side integration

The REPL's input loop now routes through `ui.prompt(...)` instead
of bare `input()`. When prompt_toolkit is available AND stdin is a
TTY AND mode is rich, the prompt gets:

- **Persistent history** across REPL launches at
  `~/.open-code/history.txt`. Up/Down arrows recall yesterday's
  prompts.
- **Autosuggest from history** -- the fish-shell ghost-text pattern.
  As you type, dim text completes from your prior commands; Right
  arrow accepts.
- **Ctrl-R reverse-i-search** through history.
- **Tab completion for slash commands** -- `/sk<Tab>` cycles through
  `/skill`, `/skills`, `/sessions`, `/schedule`, `/style`. The list
  is kept in sync with the dispatch table in `repl.py`.
- **Multi-line opt-in** via `multiline=True` (not wired into the
  main loop; available for future `/plan` / `/edit` style commands
  that need multi-line editor input).

### Fallback chain (3 levels)

1. **PT available + TTY stdin + rich mode** -> PromptSession.
2. **PT unavailable, or no TTY, or plain/json mode** -> builtin
   `input()`. Repl.py imports `readline` at module load, so on POSIX
   the fallback still has up/down history + line editing (just not
   the fancy PT features).
3. **PT initialization fails mid-flight** (Windows console quirks,
   weird PTYs, env-var inconsistencies like `TERM=xterm-256color`
   on a Git Bash session with a Windows console underneath) ->
   catches the exception, sets `_pt_available = False`, falls back
   to `input()` for THIS and all future calls.

### Bug caught during probing

The probe-driven testing surfaced a real bug. My initial
implementation wrapped only `session.prompt()` in the catch-all:

```python
self._pt_session = self._build_pt_session(history_file)  # not caught
try:
    return self._pt_session.prompt(...)                  # caught
except Exception:
    ...
```

But PromptSession construction itself can raise
`NoConsoleScreenBufferError` in Git Bash / MSYS / Cygwin terminals
on Windows -- where stdin is technically a TTY but the underlying
console isn't a Windows console buffer. The Probe Test 2 ran under
Git Bash and reproduced this exact failure. Fixed by widening the
try/except to cover the session build AND completer build AND
session.prompt(). Real-world impact: Windows users running
open-code from Git Bash now get the graceful fallback instead of
a crash on first prompt.

## Honest trade-offs

- **Slash-command completion is static.** The completer is given a
  hand-maintained list of `/foo` commands. Dynamic completion (e.g.
  `/skill rev<Tab>` -> `/skill review-pr`) would require deeper
  PT-Completer plumbing + access to skills.discover_skills + agents.
  Not done in this commit; the static list is the common-case win.
- **History is user-global, not per-CWD.** `~/.open-code/history.txt`.
  Rationale: most useful when Up-arrow gives you yesterday's prompt
  regardless of which project you're in. If a project wants its own
  history, edit and override the path in `repl.py` -- not exposed as
  a setting yet.
- **Multi-line submit is `Alt+Enter` (PT default).** Not exposed via
  a top-level argument; only available when `multiline=True` is
  passed to `ui.prompt`. The REPL's main loop currently passes
  `multiline=False`. Future enhancement: detect unbalanced brackets
  or trailing backslash and flip to multi-line automatically.
- **`prompt_toolkit` is now a hard dependency** in
  `requirements.txt`, but `ui.py` treats it as optional and
  gracefully degrades. If a user installs without PT, the
  `_try_pt() -> False` path takes over and they get builtin
  `input()` (still with readline history on POSIX).
- **Cached `_pt_available = False` is permanent for the process.**
  If PT fails once we never retry. Trade-off: avoids hammering a
  broken terminal on every prompt; means a user can't "fix" the
  terminal mid-session and have PT come back. Run a fresh `open-code`
  to retry.
