## #8 V4A apply_patch

New module `patches.py` (358 LOC). One new tool `apply_patch(patch)`
that consumes the OpenAI Codex CLI V4A envelope:

```
*** Begin Patch
*** Add File: path/new.py
+content line 1
+content line 2
*** Update File: src/existing.py
@@ def some_anchor
-    return old
+    return new
*** Delete File: tmp/old.txt
*** Move to: tmp/renamed.txt
*** End Patch
```

Supported actions: Add File / Update File / Delete File / Move
(after Update).

Hunks are anchored by **surrounding code**, not line numbers. If the
anchor matches multiple lines (and the hunk's wider context doesn't
disambiguate), the patch fails clean with an "ambiguous" error.

Path sandbox: every target path runs through `_is_under(CONFIG.cwd)`
just like `tool_write_file`. `--allow-outside-cwd` opens it.

## Honest trade-offs

- **Anchor matching uses a forgiving fallback**: if exact-match
  fails on `before` lines, we try again with both sides rstripped to
  handle trailing-whitespace mismatches. This catches the most common
  case of stale trailing spaces in patches without making the matcher
  brittle.
- **Stacked anchors (`@@ class X\n@@ def y`) work approximately**:
  the second anchor narrows the search range. Multiple-line anchor
  context is treated as "narrow then search."
- **Multiple hunks per file** apply in order against a single in-memory
  copy; we read the file once and write it once. Atomic.
- **No three-way merge**, no fuzz tolerance beyond rstrip. Stale
  patches against modified files fail clean — model retries.
- **Line endings**: we read with `encoding="utf-8"`, splitlines (which
  handles CRLF / LF), join with "\n", and preserve trailing newline.
  On Windows-native CRLF files this normalizes to LF on rewrite.
  Acceptable for v0.12; `.gitattributes` handles the canonical form.
- **The static declaration lives in `tools.py`** rather than
  `patches.py` to avoid a load-time circular import (patches.py
  imports CONFIG + _is_under from tools.py). Documented inline.
