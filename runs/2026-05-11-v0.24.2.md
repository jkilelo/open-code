## Fixes shipped

### H2 -- narrow broad `except Exception` in plugin imports

Both `output_styles.py` and `skills.py` had `try: import plugins...
except Exception: pass` wrapping the plugin scan. A bug inside
`plugins.py` (TypeError on bad JSON, AttributeError on a renamed
field) would have been silently swallowed.

Fix: split the catch into two narrower ones:
- `ImportError` for "plugins module unavailable" (truly optional)
- `OSError` for "filesystem issues during scan" (logged to stderr)

Any other exception now propagates so real bugs surface.

### H4 -- O(N^2) message-count scan -> O(1) cached counter

Before: `append_message` called `_count_messages(session)`, which
re-scanned the JSONL file from the top counting `kind:"msg"` events.
Cost: O(N) per call -> O(N^2) per session.

After: `SessionStore._msg_counts: dict[str, int]` caches the count
per session ID:
- `create()` primes the cache to 0 (no cold-scan on first append).
- `append_message()` reads from cache, increments by 1, no file read.
- `--resume` path: first `_count_messages` call does a one-time scan
  to bootstrap; subsequent calls hit the cache.

Probe `tests/probe_session_perf.py` (NEW, 4 tests):
- seq numbers stay monotonic (correctness check, not just perf)
- cache is primed at create()
- 50 sequential appends trigger ZERO read-opens of the session file
  (verified by patching `Path.open` and counting read-mode opens)
- --resume case scans exactly once then caches

Measured impact: a 200-message session previously read ~20,000 lines
of disk I/O just for `seq` numbers. Now reads 0 (or 1 on cold resume).

### H5 -- managed-settings env override renamed to test-only

The brutal review's H5: `OPEN_CODE_MANAGED_SETTINGS` env var pointed
the highest-authority settings layer at any file, creating a
priv-esc surface (a compromised subprocess could redirect managed
settings to attacker-controlled JSON before re-exec'ing open-code).

Fix:
- Renamed the env var to `OPEN_CODE_MANAGED_SETTINGS_TEST` to make
  its test-only nature explicit.
- Added a one-line stderr warning whenever it's set:
  `[managed-settings: OPEN_CODE_MANAGED_SETTINGS_TEST is set; using
  test override paths]`. A user finding it set in their environment
  notices and can investigate.
- Tests/probes updated to use the new name.

In production: only the conventional paths
(`/etc/open-code/managed.json`, `%PROGRAMDATA%\open-code\managed.json`)
are honored, removing the env-controlled priv-esc surface.
