## Fixes shipped

### Y1 -- thread `ui=ui` into `/loop` and `/schedule` callbacks

`repl.py` `_make_cb` closure now passes `ui=ui` to `run_loop`. The
callback already captures `ui` from the enclosing scope; threading
it through keeps the rich Console + status of the parent UI
consistent across loop iterations.

### Y2 -- new `ui.empty_listing(message, kind)` API

`ui.py`:
```python
def empty_listing(self, message: str, *, kind: str = "listing") -> None:
    """Emit an empty-state message that's visible in every mode."""
    if self.mode == MODE_JSON:
        sys.stdout.write(json.dumps({
            "type": "listing_empty",
            "kind": kind,
            "message": message,
        }) + "\n")
        sys.stdout.flush()
        return
    self.line(message)
```

7 call sites updated to use `ui.empty_listing` instead of `ui.line`:

| Site | Kind |
|------|------|
| `repl /sessions` empty | `sessions` |
| `repl /skills` empty | `skills` |
| `repl /agents` empty | `agents` |
| `repl /checkpoints` empty | `checkpoints` |
| `cli --list-sessions` empty | `sessions` |
| `cli --list-styles` empty | `output_styles` |
| `cli --list-plugins` empty | `plugins` |

JSON consumers now see:
```json
{"type":"listing_empty","kind":"sessions","message":"(no sessions yet)"}
```

Plain / rich callers see the human-readable message as before.

### P5 -- `reset_input()` docstring corrected

The docstring previously claimed "called after /clear so the new
session starts with a fresh in-memory history." It is NOT called by
/clear. Rewrote the docstring to document this honestly: `/clear`
deliberately preserves the PromptSession (history + autosuggest stay
warm). `reset_input` remains exposed for downstream consumers who
want a forced rebuild.

## Intentionally deferred

- **P1 -- `/restore`/`/undo` confirmation prompts use bare `input()`
  not `ui.prompt()`.** These confirmation lines (`"type 'restore'
  to confirm"`) should NOT pollute `~/.open-code/history.txt`. PT's
  cached `_pt_session` is bound to a `FileHistory` instance; reusing
  it would add "restore", "undo", "y", "n" responses to history.
  Routing them through bare `input()` is the right semantic for
  one-shot confirmations.
- **P2 -- `_rich_console()` lacks `markup=False`**, so tool output
  containing `[brackets]` renders as Rich markup. Real concern but
  pre-existing (predates v0.25.x); on the v0.26+ polish list.
- **P4 -- permission-ask prompt in `open_code.py` uses `input()`.**
  Same rationale as P1: these are one-shot per-tool confirmations,
  not free-form prompts. Polluting history would harm UX.

## Refuted hypotheses from the brutal review

- **H1 (stderr/stdout mode mismatch causing ANSI in piped stdout):**
  `model_text` default path uses `sys.stdout.write` with no ANSI.
  `markdown=True` path uses `_rich_console` whose `file=` is stderr
  (when `stderr=True`, the default). ANSI never leaks to stdout
  unless a caller passes both `markdown=True` AND constructs
  `UI(stderr=False)`. No current callsite does either.
- **H7 (double `rows` declaration in cli.main):** clean single-
  assignment with type annotation; reviewer's hypothesis wrong.
- **H15 (`table()` `max()` on empty rows generator):** the guard
  `... if rows else len(c)` short-circuits before the generator
  evaluates. No crash possible.
- **H10 (Rich markup XSS via `[brackets]` in tool output):** real
  but pre-existing (predates v0.25.x); demoted to polish P2.
