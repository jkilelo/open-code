## #10 MCP server support (stdio transport)

Hand-rolled JSON-RPC over newline-delimited JSON. **No `mcp` SDK
dependency.** ~310 LOC in a new `mcp.py` module.

### Configuration

settings.json:
```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
      "env": { "FOO": "bar" }
    }
  }
}
```

CLI flag `--no-mcp` skips startup.

### Protocol flow per server

1. **Spawn** subprocess with stdin/stdout/stderr pipes.
2. **`initialize`** -- send protocolVersion=2024-11-05, capabilities,
   clientInfo. Wait for response.
3. **`notifications/initialized`** -- fire and forget.
4. **`tools/list`** -- get the server's tool catalog.
5. Tools surfaced to the agent loop as `mcp__<server>__<tool>` in
   TOOL_DECLARATIONS.
6. When the model calls `mcp__<server>__<tool>(...)`, `run_loop`
   routes via `_MCP_CLIENT.call_tool(...)` which sends `tools/call`
   over JSON-RPC.

### Failure semantics

- Server startup failure (command not found, handshake error) logs
  to stderr but doesn't kill the session: the rest of open-code
  (built-in tools, hooks, skills) keeps working.
- `--no-mcp` is the kill switch.
- `shutdown()` terminates all subprocesses on exit (try `terminate`
  then `kill`).

## Honest trade-offs

- **stdio transport only.** No HTTP/SSE in v0.14. Adding those is
  a v0.15 candidate; stdio covers most existing MCP servers (NPM
  packages, Python tools).
- **No streaming `tools/call` responses.** The MCP spec allows
  partial results / progress notifications. v0.14 reads exactly one
  response per call; everything else is dropped. Adequate for
  bounded operations; would need work for long-running calls.
- **Hand-rolled, not the `mcp` SDK.** Zero new deps but we own the
  protocol details. If the spec evolves, we update by hand. Worth
  it for now; switch to the SDK when ready to support HTTP/SSE.
- **No timeout on `_call`.** A buggy server could hang the agent
  loop forever. Stderr is captured on EOF so the user sees what
  went wrong; we don't enforce a per-call deadline yet. v0.15+.
- **Schema translation is lossy.** We uppercase type names and
  recurse, but other JSON Schema features (oneOf, allOf, refs,
  enum) pass through unchanged. Adequate for the simple inputSchemas
  most MCP servers use.
