## What changed

`llm/_anthropic.py` -- post-coercion step in `_coerce_params` reads
`extra.cache` (constructor-level + per-call merged, per-call wins),
and when `cache.enabled` is true, anchors two `cache_control`
markers at the END of the cacheable prefix:

  1. Last tool def -- caches the entire tools block
  2. System content -- caches the system + everything before it

Two anchors stays well under Anthropic's 4-breakpoint limit. The
anchors are TERMINATORS, not boundaries, so the cached region
extends backwards from each anchor through everything that came
before -- which is exactly what we want for system prompt + tool
defs (both stable across turns within a session).

System-string -> single text block conversion is automatic. The
adapter detects `system: str` and rewrites to:

```python
[{"type": "text", "text": "<the system>", "cache_control": {"type": "ephemeral"}}]
```

ttl handling: default `"5m"`; passing `"1h"` auto-adds the
`extended-cache-ttl-2025-04-11` beta header to the params, so users
don't get a 400 from forgetting it.

Opt-in via `settings.llm.extra.cache.enabled = true`. Default is
off, because cached prompts persist on Anthropic's infrastructure
for the TTL -- some data-policies don't allow that.

## Verification

Two clients, three calls, identical 16,801-token system prompt:

```
[call 1, cached client]   in=17  cache_write=16801  cache_read=0
[call 2, cached client]   in=17  cache_write=0      cache_read=16801
[call 3, uncached client] in=16818  cache_write=0   cache_read=0
```

The system prompt was a policy paragraph repeated 400 times
(~5,600 source tokens, billed as 16,801 model tokens because of
how Anthropic counts -- the difference is the SDK's overhead +
content-block framing).

Key reading:
- **Call 2 vs call 3:** the cached call paid for 17 input tokens
  + 16,801 cache_read; the uncached call paid for 16,818 input.
  Cache_read tokens are billed at **0.1x** the normal input rate
  on Anthropic, so the cached call cost roughly 1/9 of the uncached
  one on the prefix.
- **Call 1 cache_write was 16,801, not 0.** Confirms the prefix
  was large enough to clear Haiku 4.5's 4096-token minimum on
  first call.
- **Uncached client behaviour unchanged.** Without
  `extra.cache.enabled`, the adapter doesn't touch params -- system
  stays a string, no cache_control markers anywhere, cache_write
  and cache_read both 0 on the response.

## Honest trade-offs

- **Auto-anchor placement is heuristic.** I anchor at "end of tools"
  and "end of system" -- the two most reliably stable surfaces.
  Users who want a different anchor (e.g. caching the first 10
  messages of a long conversation as well) need to bypass the
  auto-injection and place `Part.extra["cache_control"]` manually.
  v0.32+ might add `cache.anchors=["tools","system","messages_n"]`
  if real usage shows the need.

- **Opt-in, not auto-on for everyone.** I considered defaulting to
  enabled for Anthropic, but the privacy-tradeoff (prompts cached
  for 5m-1h on Anthropic's infrastructure) is real for some users.
  Easier to flip the default later than to surprise someone today.

- **Min-tokens-for-cache is silent.** If your system + tools < 4096
  tokens on Haiku, cache_control markers are accepted but cache
  hit-rate is 0. The adapter doesn't warn. Users with short prompts
  will see "enabled cache but cache_read is 0" and think it's
  broken; the README's per-model table is the only mitigation.

- **No Gemini or OpenAI counterpart yet.** Gemini has
  `cached_content` (separate API for explicit context caching) and
  OpenAI has automatic prompt caching with `prompt_cache_key`.
  Both are different mechanisms; ANY auto-injection on those
  providers needs separate design. Today's ship is Anthropic-only.
