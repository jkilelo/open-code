## UX improvements

### New `ui.py` surface

Five new UI methods that give the user feedback during work
previously done silently:

| Method | What it does |
|---|---|
| `ui.thinking(message)` | context manager that shows a Rich spinner during a long operation. No-op in plain / json modes so callers don't branch on mode. |
| `ui.model_call_start(iter, model)` | announces a model call (plain mode prints the line; rich mode is silent because the spinner shows activity) |
| `ui.autobuild_start(domain, task)` | announces the architect meta-prompt is about to fire; explicit user-facing message instead of a buried `ui.info(...)` |
| `ui.autobuild_done(name, path, tools)` | announces the new specialist with its capabilities |
| `ui.turn_summary(iters, in_tok, out_tok, wall, ...)` | concise one-line summary printed at the end of EVERY turn (not gated on `--show-metrics`) -- users always know what the turn cost |
| `ui.session_pointer(session_id, path)` | prints the session ID at the start of one-shot runs so the user can later `--resume-id` |

### Wired into the loop

- `run_loop` wraps `client.models.generate_content` /
  `_stream_iter_response` in `with ui.thinking(...)` blocks. The
  silent 2-5 second window between "calling..." and the first
  tool call is now filled with a Rich spinner.
- `_handle_request_specialist` calls `ui.autobuild_start` + wraps
  the meta-prompt LLM call in a spinner + calls
  `ui.autobuild_done` on success.
- `cli.main` calls `ui.session_pointer(...)` after creating a session.
- `cli.main` calls `ui.turn_summary(...)` after run_loop returns.

Two new lines (vs v0.26.2):

- **Session pointer at start**: `session: 6e2679e6-... (resume with --resume-id ...)`. Users now know how to come back.

In rich/TTY mode (not visible in piped output above), a spinner
animates during each iteration's `calling gemini...` window. The
spinner auto-clears when the model returns.

## Documentation

### NEW: `LEARN.md` (~720 lines)

A self-contained, copy-paste tutorial. Every example is runnable
verbatim. No "configure X first" steps -- if a command needs setup,
the setup is in the same code block.

Structure:

1. **First 5 minutes** -- 4 increasingly capable hello-worlds:
   "what's in this directory?", write+run fizzbuzz, @-file reference,
   REPL.
2. **Common patterns** -- plan-then-act, sticky permissions, JSON
   output for IDE/CI, resume, effort levels + ultrathink.
3. **The agent library** -- skills vs subagents vs autobuilt agents;
   examples of each; approval flow with `auto_approve=false`;
   versioning with `/autobuild history` + revert.
4. **Customize the system** -- OPEN_CODE.md project memory, hooks
   with example PreToolUse script, output styles, settings hierarchy
   with full example settings.json, shadow-git checkpointing + /undo.
5. **What's happening inside** -- agent loop step-by-step,
   session storage format, BM25 + embeddings hybrid rationale.
6. **10 standalone runnable scenarios** -- each is ONE command,
   no setup, covers a different real use case: code gen, security
   audit, test gen, refactor, debug, docs gen, migration, perf
   review, JSON piping, custom skill.
7. **When things go wrong** -- troubleshooting table for 8 common
   failure modes with concrete fixes.
8. **Reference** -- pointer to README, gap-log, runs/, tests/,
   file LOC table.

## Honest trade-offs

- **Spinner adds zero info; the user must trust it means "working."**
  Rich's `console.status` is a `dots` spinner. We could show
  token-arrival rate or estimated cost during streaming for richer
  feedback; deferred.
- **Turn summary is one line.** Some users want a multi-line breakdown
  by tool / by iteration. `--show-metrics` still produces the
  verbose form. The new always-on summary is concise on purpose.
- **autobuild_start fires BEFORE validation.** If the model emits
  malformed output, the user sees "building specialist for X..."
  followed by a failure log. The architect_done message only
  fires on success. The asymmetry is intentional -- failure isn't
  silent (the build-failed event is emitted + ui.warn fires) but
  it doesn't have its own ui.method.
- **LEARN.md is long (~720 lines).** Trying to be both
  progressive tutorial AND scenario reference in one file. A split
  into LEARN.md + EXAMPLES.md was considered; settled on one file
  because users in practice read top-to-bottom on first contact and
  jump-to-section on return. Section anchors are stable.
- **The 10 standalone scenarios use real-ish file paths** like
  `@README.md`. They assume the user has the open-code repo
  checked out, which is realistic for a tutorial in the repo.
- **No screenshots / animated GIFs.** Tutorial is text. Renders
  identically on every platform (consistent with the ASCII
  enforcement policy).
