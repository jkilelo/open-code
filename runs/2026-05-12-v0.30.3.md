# Run log -- open-code v0.30.3 -- 2026-05-12

**Persona:** Jeff.
**Trigger:** After v0.30.2 (Anthropic REPL working end-to-end), user
asked to flip the same settings to OpenAI and confirm parity. One
wire-level bug from OpenAI completed the multi-provider story.

## The bug

REPL on `gpt-5-mini`. Iter 1 dispatched `list_dir` cleanly. Iter 2:

```
LLMBadRequestError: openai: 400 - {'error': {'message':
'No tool output found for function call call_6ckmrTZtIQMvCZ4mfTBU5lOF.'
}}
```

OpenAI saw the `function_call` item in the input but couldn't find
the matching `function_call_output`.

## Diagnosis

The adapter's `_messages_to_native` was fine in isolation. A
diagnostic script (`tests/_debug_openai_loop.py`, since deleted)
that built the same history directly and called `ask()` twice
produced the right input items in the right order. So the adapter
worked when the caller did the right thing.

The bug was upstream, in `open_code.run_loop`:

```python
tool_content = Message(role="user", parts=tool_result_parts)
```

`role="user"` was a leftover from the Gemini-only era (Gemini sends
tool results back as user messages). The OpenAI Responses adapter
only emitted `function_call_output` items when `role="tool"`. With
`role="user"`, the inner loop checked `is_text() / is_tool_call() /
is_image()` -- none match `tool_result` -- so tool results got
silently dropped from the input.

Anthropic happened to work because its adapter processes
`tool_result` blocks regardless of role (translating `tool` -> `user`
at the end). It never noticed the missing role discrimination.

## Two-part fix

### 1. `open_code.run_loop` -- use the neutral role

```python
# WAS
tool_content = Message(role="user", parts=tool_result_parts)
# NOW
tool_content = Message(role="tool", parts=tool_result_parts)
```

The neutral type was designed with `ROLE_TOOL` exactly for this.
Each adapter maps to its own idiom:

  - Gemini: `tool` -> `user` (Gemini's tool results ride user role)
  - Anthropic: `tool` -> `user` with `tool_result` blocks
  - OpenAI Responses: `tool` -> lift `tool_result` parts to top-level
    `function_call_output` items (no wrapping `message`)
  - OpenAI Chat: `tool` -> one `{role:"tool",tool_call_id,content}`
    message per tool_result

### 2. `llm/_openai.py` -- legacy-session fallback

Old JSONL sessions (pre-v0.30.2) stored tool-result messages with
`role="user"`. After change (1), new sessions write `role="tool"`,
but `--resume` on an old session would still see `role="user"`.

Hardened the OpenAI Responses adapter's per-part loop to also lift
`tool_result` Parts out of user-role messages as top-level
`function_call_output` items. Anthropic adapter already did the
equivalent (its `_part_to_block` handles tool_result regardless of
the containing role).

## Honest lessons

- **Symmetry isn't tested by smoke checks.** v0.30.0 + v0.30.1 wire-
  tested all three adapters with hand-constructed inputs. v0.30.2 ran
  the full agent loop on Anthropic and found 6 bugs. v0.30.3 ran the
  full agent loop on OpenAI and found 1 more. Each provider taught
  the test suite something the others didn't.

- **The neutral role enum was load-bearing, but quietly so.** Until
  today, the agent loop wrote tool results with `role="user"` and
  every adapter happened to handle that. Switching to `role="tool"`
  honors the protocol's design intent and forces adapters to be
  explicit about how they translate.

- **Backward-compat fallback in the adapter is cheap insurance.**
  The 11-line "also lift tool_result out of user-role messages"
  branch in `_openai.py` costs nothing today and saves anyone who
  resumes a pre-v0.30.3 session.

## Decision

[OK] **Ship v0.30.3.** All three providers now run end-to-end
through the full agent loop, not just the adapter boundary. 49/49
probes; ASCII pure; live REPL verified on all three.
