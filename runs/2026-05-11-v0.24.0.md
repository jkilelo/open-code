## Feature shipped

### #24 /loop + /schedule

Local-to-REPL loop and delayed-execution commands. Both block the
REPL thread by design (no daemonization, no persistence) and are
interruptable with Ctrl-C.

**Usage:**
```
> /loop 30 watch git status and tell me if it changed
[/loop starting; interval=30 (30s); Ctrl-C to stop]
[/loop iter 1: watch git status...]
...

> /schedule 5m summarize what I changed in the last hour
[/schedule sleeping 5m (300s); Ctrl-C to cancel]
[(5 min later)]
[/schedule completed]
```

Duration syntax: bare seconds (`30`), or `<n>s` / `<n>m` / `<n>h`.

**New module: `schedule.py` (~110 LOC).** Public API:
- `parse_duration(s)` — `"30"`, `"30s"`, `"5m"`, `"1h"`, `"2.5m"`
  → seconds. Rejects empty, negative, bad-suffix.
- `SchedulerStats` dataclass — iterations, interrupted,
  early_stopped, total_seconds, errors[].
- `run_loop_with_interval(callback, interval, max_iterations=None,
  sleep=time.sleep)` — runs callback repeatedly. Callback returns
  False to stop early; raises KeyboardInterrupt to interrupt;
  any other exception is recorded in `stats.errors` and the loop
  continues.
- `run_schedule_delayed(callback, delay, sleep=time.sleep)` — sleeps
  then runs callback once. KeyboardInterrupt during the sleep
  cancels cleanly (iterations stays 0).

The `sleep` parameter is injectable so probes can drive the
scheduler instantly without time.sleep actually firing.

**Wiring:**
- `repl.py` `/loop <dur> <task>` and `/schedule <dur> <task>`:
  - Parse duration via `schedule.parse_duration`
  - Build a closure that drives `run_loop` once per iteration with
    the current `history` (so loop iterations share session state)
  - Dispatch to `run_loop_with_interval` or `run_schedule_delayed`
  - Print summary lines (iter count, interruption, etc.) on return.

## Honest trade-offs

- **/loop and /schedule are REPL-thread-blocking.** No daemonization
  or background scheduling. The user closes the REPL → the loop
  stops. If they want "fire and forget" they're using the wrong tool
  (cron / Task Scheduler / systemd timers). Documented in `/help`.
- **/schedule isn't persistent.** Closing the REPL between
  `/schedule` and the delayed firing cancels it. A persistent
  scheduler would need a daemon process; out of scope.
- **No exponential backoff or jitter** in `/loop`. Interval is
  literal. Watching a fast-changing dashboard at `/loop 1` will
  hammer the API. The user is responsible for picking a sane
  cadence.
- **Loop session sharing.** Each iteration writes to the SAME
  session, so the model can see prior iterations' tool outputs.
  This is intentional (it makes `/loop watch X` actually accumulate
  state) but means the JSONL grows linearly with iterations. For
  long watches, plan to `/clear` periodically.
- **`/loop` ignores model fallback signals.** If iter 1 fell back
  to a cheaper model, iter 2 still starts with `current_model` from
  the surrounding REPL state. That's fine for our use cases but
  worth noting if a future commit adds smarter cross-iter routing.
