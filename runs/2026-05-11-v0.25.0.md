# Run log -- open-code v0.25.0 -- 2026-05-11

**Persona:** Jeff.
**Trigger:** User asked for a modern terminal UI with a text-only mode.
After research (Rich + Textual + prompt_toolkit landscape, Aider's
stack), shipped the Aider pattern: Rich for output, three modes
(rich / plain / json), source stays pure ASCII.

## Research summary

The Python terminal-UI landscape in 2026:

| Library | Role | Status |
|---|---|---|
| **Rich 14.x** | output formatting (tables, panels, syntax, colors) | gold standard; used by Aider, ipdb, pip; 50K+ stars |
| **Textual** | full TUI framework (mouse, async, widgets) on Rich | overkill for a CLI |
| **prompt_toolkit 3.x** | input side (history, autosuggest, multi-line) | mature; used by Aider, IPython, bpython |
| **rich-cli / click-rich** | wrappers | scope creep |

Aider's stack (`rich` + `prompt_toolkit`) is the proven pattern for
terminal coding agents. v0.25.0 adopts the Rich half. prompt_toolkit
input-side is deferred to v0.26 if the user wants it.

## Feature shipped

### New module: `ui.py` (~285 LOC)

`UI` class wraps three modes:

- **rich** -- Rich Console with ANSI colors, Unicode box drawing,
  bold/dim styling. Default when stderr is a TTY and `NO_COLOR` /
  `OPEN_CODE_PLAIN` aren't set.
- **plain** -- pure ASCII, no ANSI escapes, no Unicode. Default when
  output is piped (`isatty()=False`) or `--plain` is passed.
- **json** -- all UI methods are no-ops; existing `_emit_json` path
  in open_code.py drives stdout. Triggered by `--print` (CONFIG.print_json).

Public methods:
- `tool_call(name, args)` -- model-issued tool invocation render
- `tool_result(name, result)` -- OK/error tag + summary line
- `status_line(**fields)` -- one-line status footer (model/iter/tokens)
- `banner(session_id, cwd, model)` -- REPL session banner (Panel in rich)
- `model_text(text, markdown=False)` -- agent's text response
- `info(msg)`, `warn(msg)`, `error(msg)` -- diagnostic lines
- `table(title, columns, rows)` -- list-of-rows (Table in rich, ASCII
  alignment in plain)

Mode detection precedence:
```
--print (json)  >  --plain  >  NO_COLOR env  >  OPEN_CODE_PLAIN env
                            >  stderr not a TTY  >  rich
```

### CLI: `--plain` flag

Added to `cli.py`. Implies plain mode regardless of TTY. Combined
with the auto-detection logic, three working invocations:

```
open-code "task"                # rich on TTY, plain when piped
open-code --plain "task"        # force plain
open-code --print "task"        # JSON (already existed)
```

`NO_COLOR=1 open-code "task"` and `OPEN_CODE_PLAIN=1 open-code "task"`
also force plain.

### Refactored: tool render call sites in `open_code.py`

5 call sites previously called `print(_render_tool_call(...))` and
`print(_render_tool_result(...))` directly to stderr. Now route
through `ui.tool_call(...)` / `ui.tool_result(...)`. The legacy
`_render_tool_call` / `_render_tool_result` functions remain as
fallbacks but are no longer used in the hot path.

The status-line emission in run_loop now goes through `ui.status_line(**fields)`.
This means rich mode displays it with dim styling, plain mode with
the existing `[k=v k=v ...]` format.

### Refactored: REPL banner in `repl.py`

`print(REPL_BANNER.format(...))` -> `ui.banner(...)`. In rich mode the
banner is a Panel with cyan border + bold/yellow/dim styled fields.
In plain mode it's the same text format that v0.4 has used.

### UI threaded through

- `cli.main` builds the UI once and passes it to `run_loop` and
  `run_repl`.
- `run_loop` accepts `ui` parameter; if None, lazily builds one via
  `UI.auto`. This keeps tests + ad-hoc callers working without
  explicit UI plumbing.
- `run_repl` accepts `ui` parameter; forwards to all internal
  `run_loop` calls (4 call sites: main loop, /plan, /act, /loop+/schedule).

## Honest trade-offs

- **Rich is now a runtime dependency.** Goes from 2 deps to 3.
  Mitigated: `rich` is pure Python, no compiled extensions, MIT
  licensed, widely deployed (Aider, ipdb, pip's tracebacks, etc).
- **prompt_toolkit deferred.** The input side (autosuggest from
  history, multi-line input, syntax highlighting in the prompt) is
  the other half of the Aider-style stack. Deliberate to keep this
  commit focused; happy to add in v0.26 if requested.
- **`_print_session_list` not yet refactored.** Listings for
  `/sessions`, `/skills`, `/agents`, `/checkpoints`, `/plugins`
  still use plain `print`. The `UI.table` method exists; routing
  the existing list renderers through it is a follow-up.
- **Rich mode in `cmd.exe` on old Windows.** Rich auto-detects
  Win10+ ANSI; very old Windows defaults to legacy color mode (uses
  fewer colors). The `--plain` flag is the always-works escape hatch.
- **Markdown rendering is opt-in.** `ui.model_text(text, markdown=True)`
  would route through `rich.markdown.Markdown`, but the default is
  plain stdout write. Rationale: tests + pipes prefer raw text;
  callers that explicitly want pretty-printed markdown opt in.
- **Source stays pure ASCII.** `probe_ascii_only` still passes. Rich
  emits Unicode at runtime; that's data, not source. The user's
  v0.24.4 promise ("100% no encoding issues in any platform") is
  preserved -- the repo itself stays 7-bit clean.

## Decision

[OK] **Ship v0.25.0.** Modern terminal UI when on a TTY; pure-ASCII
fallback when piped or when `--plain` / `NO_COLOR` requested.
44/44 probes green; 54/54 security tests green; ASCII guard still
green.
