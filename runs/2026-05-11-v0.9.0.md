## #6 Plan/Act mode separation

Builds on v0.8 #5 permission modes. Adds two REPL commands that
wrap a mode-flip around one (or two) turns and persist the plan
as a JSONL artifact.

### `/plan <task>`

1. Captures current `settings.mode`
2. Switches to `"plan"` (denies write_file + run_shell)
3. Runs the turn with the user's task
4. Reads the JSONL to find the model's final text response
5. Appends a `plan` event with: `{plan_id (UUID slug), content, model, ts}`
6. Restores prior mode
7. Prints: `[plan saved: id=<8-char>; /act to execute]`

### `/act [task]`

1. Calls `store.latest_plan(session)` to find the most recent plan
   in this session
2. Switches to `"acceptEdits"` (auto-allows write_file ask rules)
3. Builds the prompt as:
   ```
   <plan id="abc12345">
   ...plan body...
   </plan>

   <user's task here, OR a default directive: "Now execute the plan
   above. For every file the plan describes, call write_file with
   the exact path and content shown. For every shell command, call
   run_shell. Do NOT describe what you would do -- actually call the
   tools.">
   ```
4. Runs the turn

### New SessionStore methods

`sessions.py` adds:
- `append_plan(session, plan_id, content, model)` -- append `plan` event
- `latest_plan(session)` -- scan JSONL, return most recent plan event

### Lesson learned mid-implementation

First attempt at `/act` used the prompt `"{plan}\n\nExecute the
plan above."` -- the model emitted "I have successfully created
fizzbuzz.py" without calling any tools. Pure hallucination.

Fix: changed the default `/act` directive to:

> "Now execute the plan above. For every file the plan describes,
> call write_file with the exact path and content shown. For every
> shell command, call run_shell. Do NOT describe what you would do
> -- actually call the tools."

Second run produced 2 real tool calls + a concrete file on disk.
Documented in code at the `default_act_directive` constant.

## Honest trade-offs

- **`/plan` ignores tool calls from the model.** In plan mode,
  write_file and run_shell are denied by the permission layer,
  but read_file / list_dir still work. If the model wants to
  inspect a file before planning, it can. Result: plans can be
  grounded in real reads. (Acceptable; matches Cline behavior.)
- **Architect/editor model split (#9) not wired here.** The PROMPT-PACK
  noted plan/act pairs well with #9 -- use a strong model for plan,
  cheap for act. v0.9 uses a single model for both. #9 is queued.
- **`/act` directive can be overridden** by passing a task: `/act
  apply the plan but also add type hints`. When `rest` is empty,
  the default directive fires. When the user passes text, that
  text REPLACES the directive (concatenated after the `<plan>` tag).
- **The hallucination case is fixed for /act but persists for
  ad-hoc prompts.** A user typing "apply the previous plan" without
  using `/act` won't get the directive treatment. Documented.
- **Plan events accumulate without GC.** Long sessions with many
  /plan invocations leave many plan events in the JSONL. Cheap
  storage cost; would matter at 1000s of sessions. Defer.
