## #5 Permission modes

`--mode` flag + `/mode` REPL command + `settings.json` `mode:` field.
Five values mirroring Claude Code:

| Mode | Behavior |
|------|----------|
| `default` | Honor permission rules normally (deny > ask > allow) |
| `acceptEdits` | `write_file(*)` `ask` rules become auto-allow |
| `plan` | `write_file` AND `run_shell` always denied ("narrate only") |
| `auto` | Reserved for v0.9+; behaves like default for now |
| `bypassPermissions` | Skip rule evaluation entirely (hard denylist still applies) |

Layered defense intact: kernel-level deny patterns in `tools.py`
and the path sandbox are NOT bypassed by `bypassPermissions`.
This mode only skips the settings-driven `permissions.{allow,ask,deny}`
checks.

## Honest trade-offs

- **`auto` mode is a placeholder.** It evaluates the same as
  `default` in v0.8. Claude Code's `auto` mode lets the model
  request user elevation mid-session; that requires a richer
  control-flow surface and is deferred. Naming is reserved.
- **Plan mode doesn't auto-write a `plan` event** to the JSONL.
  The model's narrative response IS the plan; #6 Plan/Act adds
  the explicit `plan` event + `/act` re-execution flow.
- **`acceptEdits` only affects `write_file`** — not `run_shell`.
  This matches Claude Code's semantics (acceptEdits is about
  edits, not arbitrary shell). Users who want shell auto-allow
  should use `bypassPermissions` or explicit `allow:` rules.
- **The order matters**: mode is evaluated BEFORE PreToolUse hooks.
  So `plan` mode denies before any hook fires; the deny event still
  gets logged via `append_tool_refusal`.
