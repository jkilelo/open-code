## What shipped

### Architecture (first principles)

The model is the router. Two new tools expose the agent library:

1. `find_specialist(query, limit=5)` -- BM25-search the merged
   library of `.open-code/agents/` + `.open-code/autobuild-agents/`.
   Returns ranked matches with scores. Sub-millisecond over
   thousands of agents in pure Python.
2. `request_specialist(domain, task_example, notes)` -- run the
   architect meta-prompt to author a structured agent file, validate
   it, save to `.open-code/autobuild-agents/`, return the new name.
   Compounds: each novel domain encountered teaches the library.

System instruction now spells out the decision flow:
```
1. find_specialist(query) with keyword-rich terms.
2. If top score >= 0.6: delegate to that specialist.
3. Else if the task warrants a reusable specialist:
   request_specialist(...) then delegate to the new agent.
4. Else: answer using the base tools.
```

No separate classifier needed -- the LLM IS the classifier. This
avoids the brittleness of keyword rules and the latency of an extra
LLM round-trip.

### New modules

- `agent_search.py` (~290 LOC)
  - `tokenize()` -- lowercase, alphanumeric, drop stopwords + 1-char tokens
  - `AgentDoc` dataclass -- name/description/capabilities/domain/path/source/mtime
  - `BM25Index` -- k1=1.5, b=0.75 (Lucene defaults); inverted index;
    sub-ms search over 1000+ agents
  - `discover_indexable_agents(cwd)` -- merges both folders with
    user-shadows-autobuild precedence
  - `search_agents(cwd, query, limit, score_threshold)` -- convenience
  - Process-local mtime-keyed cache + `invalidate_cache()` for forced
    refresh

- `agent_builder.py` (~310 LOC)
  - 100+ line meta-prompt with strict markdown template:
    role / expert knowledge / workflow / output format / 2 examples /
    edge cases / refusal cases
  - `validate_generated_agent(text, existing_names)` -- parses
    frontmatter, validates required fields, kebab-case name check,
    filters allowed_tools against an allowlist (read_file/list_dir
    only -- escalation requires explicit user edit), deduplicates
    names by suffix
  - `build_agent(cwd, llm, ...)` -- end-to-end orchestration with
    `dry_run=True` option for previews
  - LLM callable is injectable -- production uses genai client;
    tests use a stub

### Safety / quality gates

- **Allowed tools allowlist** for autobuilt agents: `read_file`,
  `list_dir` only by default. Escalating to `run_shell` /
  `write_file` / `apply_patch` requires the user to hand-edit the
  generated file. Defense in depth -- a dynamically-generated agent
  doesn't get destructive power.
- **Name collision**: hand-written agents in `.open-code/agents/`
  shadow autobuild on name collision. User curation is sacred.
- **Validation before save**: malformed frontmatter, missing
  required sections (`## Role`, `## Expert knowledge`, `## Workflow`,
  `## Output format`, `## Examples`), or non-kebab-case names ALL
  reject. The model can retry.
- **Atomic deduplication**: name collision yields `-2`, `-3`, ...
  suffixes. No silent overwrites.
- **dry_run mode**: callers can preview the generated agent before
  committing to disk (used by future REPL approval flows).

## User-facing flow

User types in the REPL:
```
> how many customers purchased product x last week
```

Internally:
1. Model sees the question, decides this is SQL/analytics territory
2. Calls `find_specialist(query="sql customer purchases analytics")`
3. Result: `{"matches": [], "count": 0}` (first run, empty library)
4. Calls `request_specialist(domain="sql", task_example="how many
   customers purchased product x last week")`
5. Architect meta-prompt fires. Validates output. Saves
   `.open-code/autobuild-agents/sql-customer-analytics-agent.md`
6. Returns `{"ok": true, "name": "sql-customer-analytics-agent",
   "hint": "Now call delegate(...)..."}`
7. Model calls `delegate(agent="sql-customer-analytics-agent",
   task="how many customers purchased product x last week")`
8. Subagent runs with its specialist system prompt, produces SQL

Next session, same kind of question:
1. `find_specialist("sql customer purchases analytics")` -> top
   match score >= 0.6
2. `delegate(...)` directly. No autobuild fires. Instant routing.

## Honest trade-offs

- **One LLM call per build.** ~1k input + ~2k output tokens. Real
  cost: pennies per specialist. We don't try to cache by
  task-similarity; each `request_specialist` call independently
  authors. Caching could squeeze costs later.
- **Autobuilt agents are PROMPT specialists, not tool generators.**
  An SQL specialist doesn't auto-create a `run_sql` tool. It writes
  excellent SQL; the user runs it. This is the right scope -- adding
  dynamic tool creation is a security can of worms that needs its
  own design pass.
- **BM25 is bag-of-words, no semantics.** A query like "find slow
  queries" won't match an agent described as "query performance
  analysis" unless terms overlap. Future polish: optional embedding
  rerank. BM25 is the right baseline -- fast, deterministic,
  explainable.
- **Stopword list is conservative.** Just 24 obvious English words.
  Cross-language search won't work. Reasonable for English-first
  agent descriptions.
- **No quality scoring of generated agents over time.** A bad agent
  built once persists until manually deleted. Future: agent
  versioning + thumbs-up/down from user.
- **The architect uses the same model as the main loop by default.**
  Set `settings.models.architect` to use a different (more capable)
  model for the meta-prompt step.
- **The autobuild folder is committed-to-git by default.** Users who
  want different behavior add `.open-code/autobuild-agents/` to
  `.gitignore`. Reasoning: agents the system builds for you ARE part
  of the project's intellectual capital and benefit from version
  control.

## Why this is "day one valuable"

Within the first session:
1. User asks an SQL question. Specialist built. SQL answered well.
2. User asks a scraping question. Specialist built. Scraping answered well.
3. User asks an ML question. Specialist built. ML answered well.

By turn 4, the library has 3 specialists. The 4th question
("can you tune that SQL query") hits the SQL specialist
immediately -- no rebuild. The system gets sharper with use.

The user did nothing special. They asked questions. The library
grew. The next user inheriting the same project gets all those
specialists for free.
