## Per-blocker evidence

### Blocker 1 -- Denylist coverage

**Before:** 20/25 adversarial commands bypassed (review found `rm -r -f /`, `Remove-Item -rf C:\Users`, `rd /s`, `git push --force`, `curl ... | sh`, `> /etc/passwd`, etc. all sailed through).

**After:** `tests/probe_denylist.py` runs the brutal review's full adversarial set of 30 commands. Re-baselined:

```
$ python tests/probe_denylist.py | grep -c CAUGHT
30
$ python tests/probe_denylist.py | grep -c MISS
0
```

Approach: regex patterns for unambiguous strings (`shutdown`, `mkfs`, `format C:`, `git push --force`, etc.); two token-aware helpers for `rm` and `Remove-Item / ri / rmdir` that detect "both recursive and force flags in any form" -- handles combined `-rf`, separated `-r -f`, long `--recursive --force`, sub-shell wrappers (`sh -c '...'`), pipelines (`ls | xargs rm -rf`), and full-path invocation (`/usr/bin/rm`).

False-positive trade-off: `echo 'a > /etc/passwd'` (quoted, not actually redirecting) is over-blocked. Documented and accepted -- `--allow-dangerous` opts out, and over-blocking on /etc/passwd writes is safer than under-blocking.

`tests/test_security.py`: 54/54 pass -- includes 38 dangerous-blocked + 12 safe-allowed + 4 path-sandbox cases.

### Blocker 2 -- `--resume` history cap

**Before:** `probe_resume_bloat.py` measured ~101k tokens loaded after 200 turns in one CWD, no pruning anywhere in source.

**After:** `session_resume_for_cwd(conn, cwd, max_messages=80)` trims to the last N messages, with the additional rule that the history must begin on a user turn (Gemini API requires it; trim forward if the cap landed mid-exchange).

```
$ python tests/probe_resume_bloat.py
[max=0]   loaded 400 msgs (101545 est tok), dropped=0
[default] loaded 80 msgs (20320 est tok), dropped=320
[max=10]  loaded 10 msgs (2540 est tok), dropped=390
OK -- resume cap working.
```

CLI flag `--resume-max-messages` exposes the override (also `OPEN_CODE_RESUME_MAX` env var). When the cap drops messages, stderr prints `[resuming session 1 -- 80 prior messages; 320 older message(s) dropped to keep input bounded (raise --resume-max-messages or pass 0 to disable)]` so Jeff knows what's happening.

### Blocker 3 -- Model fallback chain

**Before:** Preview model 404 -> fatal error.

**After:** `MODEL_FALLBACK_CHAIN` constant + `_is_model_unavailable_error()` classifier + retry loop in `run_loop`. On the first availability-class error, drops the failed model and retries the same iteration under the next candidate. Subsequent iterations use the working model.

Live probe (`tests/probe_fallback.py`):

```
[OK ] msg='404 Not Found: models/foo'  want=True got=True
[OK ] msg='model not found'  want=True got=True
[OK ] msg='Model is deprecated'  want=True got=True
[OK ] msg='invalid model name'  want=True got=True
[OK ] msg='This model is unavailable'  want=True got=True
[OK ] msg='rate limit exceeded'  want=False got=False
[OK ] msg='permission denied'  want=False got=False
[OK ] msg='quota exceeded'  want=False got=False
[OK ] msg='authentication failed'  want=False got=False
[OK ] msg='Connection timeout'  want=False got=False

=== Live fallback with bogus primary ===
PONG
exit_code=0 final_model='gemini-3.1-flash-lite'
OK -- fell back from bogus -> 'gemini-3.1-flash-lite'
```

Conservative classifier -- only "model not available" signals trigger fallback. Quota/auth/network errors still fail loudly. The fallback message goes once to stderr, never to stdout.

End-to-end PowerShell live test:
```

[iter 1] calling gemini-nonexistent-99...
```

The `metrics["model"]` updates to the working fallback, so `--show-metrics` reports the actual model used, not the requested one.
