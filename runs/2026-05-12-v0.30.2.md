## Six bugs surfaced by actually running the REPL on Anthropic

The pattern: I shipped v0.30.0 and v0.30.1 by running adapter-level
smoke checks (`tests/_live_*_check.py`), which exercised the protocol
surface but bypassed the full `run_loop`. Anything *between* the CLI
entry and the adapter's first method call was untested for non-Gemini
providers. Today's REPL run found six such gaps.

### 1. `cli.py` passed the Gemini key to whichever provider was configured

```
LLMAuthenticationError: anthropic: 401 - {'message': 'invalid x-api-key'}
```

Root cause: `cli.main` reads `GEMINI_API_KEY` early and threads it
through `_make_llm_from_settings(settings, api_key)`. The factory
then passes it as the provider's API key. So even with
`provider="anthropic"`, the Gemini key got sent to Anthropic as
`x-api-key`.

Fix in `open_code.py:_make_llm_from_settings`: only honor the
caller's `api_key` when `provider == "gemini"`. For other providers,
pass `api_key=None` so the factory falls back to its own
`DEFAULT_API_KEY_ENV[provider]` env lookup.

### 2. `settings.llm.model` was ignored

REPL banner showed `Model: claude-haiku-4-5` (settings.llm picked
up) but the actual call used `gemini-3.1-flash-lite-preview` (CLI
default).

Root cause: `cli.py`'s model resolution only checked the legacy top-
level `settings.model`, not the new nested `settings.llm.model`. So
the model passed into `run_loop` was always the Gemini CLI default.

Fix: extended the resolution chain to:
`--model` -> `OPEN_CODE_MODEL` env -> `settings.llm.model` ->
`settings.model` (legacy) -> `DEFAULT_MODEL`.

### 3. REPL banner hardcoded "Gemini coding agent"

`repl.REPL_BANNER` and two branches in `ui.banner()` had literal
"Gemini coding agent (REPL mode)" strings. Confusing when running on
Anthropic.

Fix: all three -> "LLM-agnostic coding agent (REPL mode)". Future
work: show the actual configured provider in the banner.

### 4. `TOOL_DECLARATIONS` used UPPERCASE Gemini-flavored schema types

```
LLMBadRequestError: anthropic: 400 - tools.0.custom.input_schema.type:
Input should be 'object'
```

Root cause: 5 of the 7 built-in tool declarations (`read_file`,
`write_file`, `list_dir`, `run_shell`, `apply_patch`) plus
`subagents.DELEGATE_TOOL_DECLARATION` and `mcp._normalize_schema()`
all emitted `"type": "OBJECT"` / `"STRING"` / `"INTEGER"`. Gemini's
SDK accepts both cases; Anthropic and OpenAI reject anything but
lowercase JSON Schema.

Fix: normalized all 11 sites in `tools.py`, `subagents.py`, `mcp.py`
to lowercase JSON Schema types. `mcp._normalize_schema()` now keeps
types lowercase (previously translated to UPPERCASE for Gemini).
`tests/probe_mcp.py` assertions updated to match.

### 5. `tool_call_id` wasn't carried from `tool_use` to `tool_result`

```
LLMBadRequestError: anthropic: 400 -
messages.2.content.0.tool_result.tool_use_id: String should match
pattern '^[a-zA-Z0-9_-]+$'
```

Root cause: the model returns a `tool_use` block with `id="toolu_..."`
which the Gemini/Anthropic/OpenAI adapter captures into
`Part.tool_call_id`. But the agent loop's dispatch then calls
`Part.make_tool_result(name, result)` -- no `tool_call_id` argument.
Result Part has empty string id; Anthropic regex
`^[a-zA-Z0-9_-]+$` requires at least one char, rejects.

Fix in `open_code.run_loop`: capture `call_id = fc.tool_call_id`
once per iteration's function_calls loop. All 5
`Part.make_tool_result(name, result)` sites in the dispatch (deny,
ask-declined, hook-block, normal completion, plus 2 in REPL ask
paths) now pass `tool_call_id=call_id`.

### 6. Windows cp1252 console couldn't render Claude's emoji output

```
UnicodeEncodeError: 'charmap' codec can't encode character '\u2B50'
```

Root cause: the model emitted a star emoji in its response.
Python on Windows defaults to cp1252 for redirected stdout (PowerShell
pipe), which can't encode U+2B50. Generic Python-on-Windows quirk,
exposed now because Claude formats output with more decoration than
Gemini.

Fix in `open_code.py` preamble: reconfigure `sys.stdout` and
`sys.stderr` to UTF-8 with `errors='replace'` if `.reconfigure()` is
available. Idempotent; no-op on POSIX (already UTF-8) and on
TextIOWrappers that don't expose `.reconfigure`.

## Honest lessons

- **"Adapter smoke-test green" != "agent loop works."** The smoke
  scripts (`_live_*_check.py`) bypass `cli.py` and `run_loop` entirely;
  they hit `make_llm_client` -> adapter directly. Five of today's
  six bugs were in the cli/run_loop layer, not the adapter. v0.30.0
  shipped with the adapter wired correctly and the agent loop's
  multi-provider hooks half-wired -- regression coverage at the
  smoke level missed all of it.

- **The fix:** treat "REPL works on each provider" as a release
  criterion, not "smoke tests pass." Future v0.30.x adds an
  integration probe that piped-input-runs the REPL against a fake
  LLMClient for each provider's tool-call/result shape.

- **`_make_llm_from_settings` was load-bearing in a way I missed.**
  The "pass legacy api_key for Gemini back-compat, let factory
  resolve for others" rule should have been there in v0.29.0. Fixing
  it as a v0.30.2 patch is the right move, but the design intent
  needs the comment block I just added.
