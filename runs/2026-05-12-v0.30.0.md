## What changed

Replaced the flat v0.29.0 `llm.py` + `llm_gemini.py` with a `llm/`
package and three provider adapters. The agent loop's `LLMClient`
interface is now backed by a real abstraction: a shared base class
with error translation + stop-reason normalization + a streaming
accumulator, plus per-provider files that handle only the deviations.

### New package layout

```
llm/
+-- __init__.py           # public re-exports (back-compat: `from llm import X`)
+-- types.py              # Part (5 kinds), Message, ToolDecl, Usage,
+--                       #   AskResult, StreamChunk
+-- errors.py             # 12-class hierarchy
+-- protocol.py           # LLMClient Protocol (structural)
+-- base.py               # BaseLLMClient ABC + StreamAccumulator helper
+-- factory.py            # make_llm_client + DEFAULT_MODELS
+-- _gemini.py            # google-genai 2.1.0 adapter
+-- _anthropic.py         # anthropic 0.101.0 adapter
+-- _openai.py            # openai 2.36.0 Responses API adapter
+-- _openai_chat.py       # openai 2.36.0 Chat Completions (OSS-compat)
```

### Shared base ("overlaps" -- the user's word)

`BaseLLMClient` provides:

- **Error translation**: provider-native exceptions are caught at the
  `ask`/`ask_stream`/`embed` boundary and mapped to the neutral
  `LLMError` tree. Subclasses override `_translate_error` for
  isinstance-checked precision; the base falls back to name-based
  heuristics so even unseen exceptions land somewhere sensible.
- **Default `embed()`** that raises `LLMConfigError` with the "use
  Voyage / OpenAI / Gemini for vectors" hint -- only Anthropic
  actually exercises this path.
- **`normalize_stop_reason(...)`**: one table maps Gemini's
  `MAX_TOKENS`, Anthropic's `max_tokens`, OpenAI Chat's `length`, and
  OpenAI Responses' `incomplete` all to neutral `"length"`. Eleven
  finish-reason values from three providers collapse to seven
  neutral codes.

`StreamAccumulator` provides:

- One state machine for `add_text` / `add_thinking` / `set_signature`
  / `start_tool_call` / `add_tool_call_args` / `set_usage` /
  `set_stop_reason` calls.
- `build_message()` synthesizes the final `Message` with parts in
  Anthropic's required order (thinking -> text -> tool_calls).
- Tool-call args buffered as partial-JSON strings (matches Anthropic
  `input_json_delta` and OpenAI's chunked `function_call_arguments`),
  parsed on `build_message`. Malformed JSON falls back to
  `{"_raw": "..."}` so an SDK bug never crashes the agent loop.
- `set_usage` (replace) for cumulative deliveries (Anthropic
  `message_delta`); `add_usage` (sum) for incremental.

Net: each adapter's streaming code is ~50 lines of "translate this
event type -> call this accumulator method" rather than ~150 lines of
buffer management.

### Per-provider files ("deviations")

What each adapter actually does that's *not* shared:

- **`_gemini.py`**: thinking config branches on model family (3.x
  uses `thinking_level` enum; 2.5 uses `thinking_budget` int with
  Pro/Flash budget tables). `thought_signature` is captured into
  `Part.extra` on EVERY part type (not just function_calls -- Gemini
  3 puts them on text parts too). AFC explicitly disabled. Safety
  blocks raised as `LLMSafetyError` despite the SDK returning HTTP
  200 with `finish_reason==SAFETY`.

- **`_anthropic.py`**: `max_tokens` defaults to 4096; auto-bumped to
  16384 when thinking is enabled (must exceed budget_tokens or the
  API rejects). `temperature` clamped to 0..1 (Anthropic's range).
  Tool definitions use `input_schema` (not `parameters`). Tool
  results carry `tool_use_id` for matching parallel calls. Adaptive-
  thinking models (Opus 4.6+, Sonnet 4.6, Haiku 4.5, Opus 4.7) get
  `thinking={"type":"adaptive"}`; older ones get
  `{"type":"enabled","budget_tokens":N}`. Cache control passed
  through via `Part.extra["cache_control"]`. `embed()` raises
  `NotImplementedError` -> base maps to `LLMConfigError`.

- **`_openai.py`** (Responses API): flat tool shape (no nested
  `function:` wrapper). `tool_calls` are top-level `function_call`
  items, not nested in messages. Tool results are
  `function_call_output` items. Reasoning items with
  `encrypted_content` round-trip via `Part.extra` when `store=False`.
  No `top_k`/`seed`/`stop_sequences`/`presence_penalty`/
  `frequency_penalty` support -- silently dropped per the lenient
  default. `xhigh` effort level is the highest.

- **`_openai_chat.py`** (Chat Completions): nested `function:`
  wrapper on tools. `tool_calls` ride on the assistant message.
  Tool results are `{"role":"tool","tool_call_id":...,"content":...}`
  messages. `stream_options={"include_usage": True}` opt-in for
  usage in stream. `base_url` honored via `extra={"base_url":...}`
  so Groq/vLLM/Anthropic-OpenAI-shim work via `provider="openai_chat"`.

### Extended neutral types

`Part` now has 5 kinds (was 3): text / tool_call / tool_result /
**thinking** / **image**, plus new fields:

- `tool_call_id: str` -- unified across providers (Anthropic
  `tool_use_id`, OpenAI `call_id`, Gemini's optional `FunctionCall.id`)
- `is_error: bool` -- auto-detected from `{"ok": False}` for
  open-code's tool convention; explicit `is_error=True` also
  honored
- `image_mime` / `image_data` / `image_url` / `image_file_id` --
  one Part shape for all three providers' image inputs

`Usage` gained `reasoning_tokens` (OpenAI o-series, Gemini thoughts,
Anthropic thinking) and `cache_write_tokens` (Anthropic
`cache_creation_input_tokens`).

### LLMClient.ask signature

22 first-class kwargs covering everything common across the three
providers:

```
model, messages, system_instruction, tools, tool_choice,
parallel_tool_calls, temperature, top_p, top_k, max_output_tokens,
stop_sequences, seed, presence_penalty, frequency_penalty,
candidate_count, metadata, user, thinking_effort, thinking_budget,
include_thinking, response_schema, response_mime_type,
extra={}        # provider-specific escape hatch
```

`thinking_effort` (string: `off`/`minimal`/`low`/`medium`/`high`/
`max`) is the neutral knob; adapters map to native shapes. Pass
`thinking_budget` (int) if you want explicit Gemini 2.5 / Anthropic
enabled-mode budgets; adapters auto-pick for adaptive models.

### Storage updates

`sessions.py` extended for the new Part fields:

- `function_call` entries gain `call_id` and `extra` (already added
  in v0.29.0 -- now also persists `tool_call_id`)
- `function_response` entries gain `call_id` and `is_error`
- New `thinking` entry: `{"type":"thinking", "text":..., "extra":...}`
- New `image` entry: `{"type":"image", "mime":..., "data_b64":...,
  "url":..., "file_id":...}` -- bytes base64-encoded for JSON safety

Old JSONL files (pre-v0.30) round-trip cleanly: missing fields
default to empty-string/False, missing entry types are skipped.

## Honest trade-offs

- **No Anthropic / OpenAI live tests.** I don't have keys for those
  in this environment. The adapters compile, import lazily, and
  pass their unit-level translation tests in probe_llm. First real
  call against the wire will probably surface a quirk or two -- the
  adapter shapes are based on docs research, not empirical
  verification. Recommended next step: a tiny `live_anthropic.py`
  smoke + same for OpenAI.

- **OpenAI Responses encrypted_content round-trip is untested.**
  When `store=False`, reasoning items must replay with
  `encrypted_content` intact. Adapter captures + restores via
  `Part.extra["encrypted_content"]` but live verification needs an
  OpenAI key.

- **Anthropic streaming `usage` cumulative-vs-incremental gotcha**
  is handled in StreamAccumulator (`set_usage` replaces). If a
  future Anthropic SDK change makes it incremental, that's one
  method swap away.

- **Multimodal coverage is one-way only.** Open-code can SEND
  images via the new `Part.make_image(...)` constructor, but the
  CLI doesn't yet expose a way to attach images at the prompt
  layer. Open-code's `@file` provider would be the natural seam.
  Deferred to a future PR.

- **Async is sync.** All three SDKs have async clients (`client.aio.*`,
  `AsyncAnthropic`, `AsyncOpenAI`). Protocol exposes only sync
  methods. `ask_async` / `ask_stream_async` is a natural additive
  v0.31 hook.

- **No automatic prompt caching.** Anthropic users can pass
  `Part.extra["cache_control"]={"type":"ephemeral"}` manually but
  open-code doesn't auto-place breakpoints. Anthropic's cost model
  makes this attractive for system-instruction-heavy flows; v0.31
  candidate.

- **Pyright noise**: every cross-module import shows as Unknown
  because there's no `pyrightconfig.json` for the open-code source
  root. Non-blocking; runtime is clean.

## Decision

[OK] **Ship v0.30.0.** Three-provider LLM-agnostic layer with
properly shared base class for overlaps (streaming accumulator,
error translation, stop-reason normalization) and per-provider
adapters for deviations (Gemini's thought_signature on every part,
Anthropic's adaptive vs enabled thinking, OpenAI's Responses-vs-Chat

across one-shot + tool-call + resume.
