## Two real blockers closed

### 🔴 → ✅ MCP `_call` concurrency loss-of-response

**Bug** (`mcp.py:196-197` and `:223-225` in v0.14.1):
- `msg_id = srv.next_id; srv.next_id += 1` was unlocked. Two threads
  could claim the same id.
- The single-Queue design silently dropped messages whose id didn't
  match the current caller, but those were responses other waiters
  needed.

**Reviewer's evidence**: 8 parallel `_call`s → only 3/8 completed in
15s; the rest hung to timeout.

**Fix**: per-request `threading.Event` + per-msg-id response slots,
guarded by `response_lock`. The reader thread dispatches each parsed
message directly to the right waiter's slot+event. `next_id` allocation
guarded by `next_id_lock`. Plus an EOF wake-all signal so dying
servers don't strand waiters until their individual timeouts.

**Probe** (`tests/probe_mcp_concurrency.py`):
```
[6] ok content=echo 9
[7] ok content=echo 10
final next_id after 8 concurrent calls + 2 init calls = 11
reader thread alive before shutdown: True
reader thread alive after shutdown(3s wait): False
[PASS] reader thread exited after shutdown
[PASS] all 8 concurrent calls completed without exceptions
```

8/8 calls now complete. `next_id` advances monotonically. Reader
thread shuts down cleanly within 3s.

### 🔴 → ✅ Hook RCE-by-cd-into-hostile-repo

**Bug**: hooks under `.open-code/hooks/` fired automatically on first
event. No allowlist, no consent prompt. Cloning a hostile repo and
running open-code there executed attacker code.

The brutal reviewer was emphatic this is NOT 🟡:
> "Calling RCE-by-cd-into-hostile-repo a 🟡 because the fix is 'v0.15
> candidate' is the calibration failure the brutal-reviewer warned
> about. Anyone telling a user 'Tier 1 is honestly complete' should
> not include unmitigated arbitrary code execution from clone-and-run
> as a tolerable gap."

**Fix**: trust gate at session start, modeled on VS Code's Workspace
Trust pattern:
- `~/.open-code/trusted-projects.json` persists per-project decisions
- `hooks.ensure_hooks_trusted(cwd, interactive=..., trust_override=...)`
  is called once by `cli.main` before any hook fires
- **Interactive TTY**: user is shown the inventory of hook scripts and
  chooses `allow once / trust always / deny`. "Trust always" persists.
- **Non-interactive** (piped one-shot): **auto-deny** unless
  `--trust-hooks` was passed. Stderr explains how to opt in.
- `hooks.fire()` consults the in-process trust cache before invoking
  any script. Untrusted projects → empty `HookResult`, no execution.
- Trust is keyed by the **project root** (parent of `.open-code/`),
  so running open-code from a subdir of a trusted repo still works.
- New CLI flags: `--trust-hooks` (allow once) and `--no-hooks`
  (disable entirely for this invocation).

**Probe** (`tests/probe_hook_security.py` — inverted from the v0.14.1
exploit-demo):
```
[PASS] untrusted project: hook refused; marker NOT written
[PASS] trusted project: hook fires; marker written: 'hello from your hooks'
[PASS] non-interactive mode auto-denies untrusted project
[PASS] trust_override (--trust-hooks) bypasses non-interactive auto-deny

OK -- hook trust gate works end-to-end. RCE-by-cd-into-hostile-repo blocked.
```

The same script that demonstrated the exploit in v0.14.1 now
demonstrates the fix. Adversarial PreToolUse hook doesn't write its
marker until the user explicitly trusts the project.

### Bonus: stderr drainer in mcp.py

**Concern** the reviewer flagged: `subprocess.PIPE` for stderr with
no reader thread → a chatty MCP server can fill the stderr buffer
(~8-64 KB), block on its own stderr write, never reply on stdout,
surface as a timeout.

**Fix**: second daemon thread per server drains stderr into a bounded
1000-line buffer (`srv.stderr_buf`). Used for diagnostics in the EOF
error path. No more pipe-buffer deadlock.

## Cultural lesson

The reviewer was sharp on this:
> "The lesson didn't transfer between modules — the maintainer fixed
> `patches.py` with collect-all-then-check-ambiguity discipline but
> wrote `mcp.py` with collect-one-then-discard-the-rest discipline."

I treated each fix as isolated. The pattern (`collect_one_then_discard_rest`)
was the underlying bug class — I only fixed the named instance in
`patches.py` without applying the same discipline to `mcp.py`.

The v0.14.2 fix uses **per-recipient dispatch with explicit registration**
(threading.Event + slot dict). Not just "collect all and check
ambiguity" — that wouldn't work for MCP because callers genuinely
need to receive a specific response, not detect that multiple
arrived. The right discipline is: when there's a 1:N producer→consumer
relationship, dispatch by key, never broadcast-and-filter.
