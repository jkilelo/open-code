## Feature shipped

### #25 Managed settings enforcement

A new "managed" settings layer that's read AFTER user / project /
project-local and therefore overrides per-user choices. Intended for
org-wide enforcement of policy: forced deny rules, forced
`hooks.disabled`, locked model, etc.

**Paths:**
- POSIX: `/etc/open-code/managed.json`
- Windows: `%PROGRAMDATA%\open-code\managed.json` (default
  `C:\ProgramData\open-code\managed.json`)
- Env override: `OPEN_CODE_MANAGED_SETTINGS` (colon-separated on
  POSIX, semicolon on Windows) — primarily for tests and per-host
  variations, but useful for kiosk deployments too.

**Merge semantics:**
- New layer order: user → project → local → managed.
- `_merge` deep-unions lists (unchanged): managed `deny` rules
  ADD to existing project deny rules — they don't replace them.
- Scalars (model, hooks.disabled, etc.) are overwritten last-wins,
  so the managed file's value wins.
- `permissions.deny > always_allow > ask > allow` order is unchanged,
  so a managed `deny` still beats a managed `always_allow`. The
  admin can't accidentally weaken their own deny rule by also
  granting always-allow.

**Why this matters:**
- An enterprise can ship a `/etc/open-code/managed.json` with
  `permissions.deny = ["run_shell(curl * | sh)"]` and a
  `hooks.disabled: true` and have it apply across all users on a
  shared workstation. The user's `~/.open-code/settings.json`
  can't unset it.

## Honest trade-offs

- **No validation of `managed.json` schema.** A malformed managed file
  is silently skipped by `_load_one` (returns None) rather than
  refusing to start. Trade-off: prefer "open-code keeps working with
  policy partially applied" over "open-code refuses to start because
  of an admin typo." Worth a future `--strict-managed` flag.
- **The env var `OPEN_CODE_MANAGED_SETTINGS` lets a user bypass the
  managed file** if they can set their own env. This is by design:
  managed settings can't enforce against root-equivalent attackers,
  and for power users the override is convenient for testing. The
  enterprise threat model assumes the admin controls the env.
- **No per-tool-key locking.** A user can still hand-edit
  `settings.local.json` to set `model` to something else; the
  managed layer overrides their `model` at LOAD time but doesn't
  remove it from disk. Next: a "locked keys" field in the managed
  file that prevents downstream layers from setting them at all.
  (Not done in this version.)
- **Path defaults are conventional but not standardized.** Real
  enterprise tooling might want `/etc/open-code/managed.json.d/*.json`
  drop-in style. Acceptable v1.
