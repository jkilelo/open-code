## What the user sees

**On a TTY (rich mode):** Tables get a Rich `Table` widget with a
bold-cyan header, automatic column sizing, and alternating-row
hints from Rich's default theme. The title prints as a centered
heading above the table.

**Piped / `--plain` / `NO_COLOR`:** Plain ASCII columns with `--`
separator lines. The existing `ui.table` plain-mode implementation
(introduced in v0.25.0) computes column widths from the data so
every column lines up. Verified by extending `probe_ui.py`'s
Test 2 (plain mode all-methods sweep).

**`--print` (JSON):** `ui.table` is a no-op, same as every other
UI method. JSON consumers don't see human-readable listings.

## Kept for back-compat

The original render functions still exist:
- `skills.render_skill_listing(skills) -> str`
- `subagents.render_agent_listing(agents) -> str`
- `plugins.render_plugin_listing(plugins) -> str`

None of the v0.25.2 hot paths use them, but they're exported and
some external tooling (or future probes) might depend on them.
Removed-vs-kept call: keeping them is cheap, removing them risks
breaking unseen callers.

## Honest trade-offs

- **No new probe.** The refactor reuses existing `ui.table` which
  is already probe-asserted (pure ASCII in plain, ANSI in rich,
  no-op in JSON). Adding REPL-integration probes would require a
  TTY harness; not worth it for a call-site rename.
- **Title text duplication.** Each call site supplies its own
  title string. A future polish could centralize them via a
  `LISTINGS` registry, but for 7 call sites the duplication is
  fine.
- **Description truncation lengths vary** (40 / 55 / 37 chars
  depending on the listing). Carried over from the original
  hand-rolled strings; not unified. Future polish: a
  `ui.table(..., max_col_width=...)` argument would let Rich auto-truncate.
- **`_print_session_list` is now unreferenced in the hot path.**
  Kept as exported API in case external tooling imports it. The
  v0.4.0 contract (importable from `open_code`) is preserved.
