# Run log -- open-code v0.27.2 -- 2026-05-12

**Persona:** Jeff.
**Trigger:** User ran v0.27.1 in their PowerShell terminal for the
first time. The panel rendered the cyan border + row 1 then
**everything else vanished** -- no row 2, no row 3, no closing
border, no model response, no tool calls, no turn summary. Clean
exit back to prompt with no error.

That's a real bug. v0.27.1's panel ate the agent's output.

## Root cause

`ui.LiveStatusPanel.start` constructed Rich's `Live` with:

```python
self._live = Live(
    ...
    redirect_stdout=False,
    redirect_stderr=False,
)
```

Rich's `Live` defaults to `redirect_stdout=True, redirect_stderr=True`
for a critical reason: while the live area is rendered at the
bottom of the terminal, any `sys.stdout.write` / `sys.stderr.write`
call needs to be routed through Rich's Console so it can be
buffered and rendered ABOVE the live area on the next refresh.

With redirect=False, those writes hit the raw file descriptor at
the cursor's current position -- which Live owns -- and get
overwritten on the next refresh cycle. The streamed model text
(`sys.stdout.write(text)` inside `_stream_iter_response`), the
tool render lines (which still go through ui.tool_call but
ultimately reach Rich), and the various stderr writes for
iteration headers / errors / checkpoint messages: all destroyed.

The agent loop completed correctly in memory -- model_content was
built, function calls dispatched, results collected -- but the
visual representation was annihilated.

## Why probes didn't catch it

In the test environment, stderr is not a TTY, so the panel returns
`NoOpPanel`. The probes assert this graceful-degradation contract.
But none of them exercise the actual rich-rendered code path,
because that requires a real terminal harness (which we don't have
in CI).

This is a class of bug only a live run can surface. The user
caught it on first contact -- exactly when it matters most.

## Fix

`ui.LiveStatusPanel.start` now uses the Live defaults (redirect=True):

```python
self._live = Live(
    self._render(),
    console=self._console,
    refresh_per_second=4,
    transient=True,
)
```

With redirect=True:
- Streamed model text goes through Rich, renders cleanly above panel
- Tool call/result prints from ui.* go through Rich (already did)
- stderr writes for errors / iteration headers go through Rich
- `_emit_json` writes to stdout are irrelevant here (JSON mode auto-
  disables the panel via `NoOpPanel`, so there's no conflict)

## Regression probe

Added to `probe_ui.py`:

```python
import inspect
panel_src = inspect.getsource(UI_MOD.LiveStatusPanel.start)
assert "redirect_stdout=False" not in panel_src, (
    "v0.27.1 regression: Live(redirect_stdout=False) eats streamed "
    "model text + tool prints. Defaults must be used (True)."
)
assert "redirect_stderr=False" not in panel_src
```

This is a source-inspection assertion (similar to the B2 path-safety
guard from v0.26.2). A refactor that re-disables redirects would
trigger the probe. It's not a full live-rendering test (we can't do
that in CI) but it locks the specific config choice.

## Honest trade-offs

- **The bug was a wrong-default choice.** Rich's Live defaults are
  carefully designed for this exact case (sticky footer + scrolling
  output). I overrode them to be cautious about `_emit_json` writes,
  but `_emit_json` only fires in JSON mode which auto-disables the
  panel -- so there was no actual conflict to defend against. The
  override was premature optimization that broke the feature.
- **Source-inspection probe is brittle.** A renamed local variable
  could break it. But the trade-off vs no test at all is clear.
  Real live-rendering tests require a pseudo-TTY harness; out of
  scope.
- **First-contact bug recovery loop took 2 iterations.** v0.27.1
  shipped Friday; v0.27.2 ships Friday after a live run. That's the
  brutal-review pattern paying off: real usage catches what probes
  can't.
