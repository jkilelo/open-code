## What changed

### New: `lsp.py` (~600 LOC)

Self-contained LSP client following the same shape as `mcp.py`:

- `LSPServer` dataclass: per-language state (proc, response slots,
  diagnostics cache, open-doc versions, stderr buffer)
- `LSPClient`: lazy-spawn per language; routes path-based queries
  to the right server via `file_patterns` matching
- Content-Length-framed JSON-RPC over stdio (bytes mode, not text --
  LSP framing needs raw bytes for the `Content-Length: N\r\n\r\n`
  header). Differs from MCP, which uses newline-delimited JSON.
- Per-server reader thread dispatches by message id (response) or
  method+URI (publishDiagnostics notifications). Stderr drainer
  prevents pipe-buffer deadlock on chatty servers.
- Polite shutdown: `exit` notification + terminate + 2s wait + kill.

### Four new agent tools (`tools.py`)

| Name | Maps to |
|---|---|
| `lsp_diagnostics(path)` | `textDocument/didOpen` + waits for `publishDiagnostics` |
| `lsp_hover(path, line, col)` | `textDocument/hover` |
| `lsp_definition(path, line, col)` | `textDocument/definition` |
| `lsp_references(path, line, col)` | `textDocument/references` (with `includeDeclaration`) |

All four return `{ok, ...}` dicts. If LSP isn't configured or no
server matches the file's language, they return ok=False with a
helpful error -- never crash the agent loop.

Tool declarations note that lines/cols are 0-indexed (LSP spec) so
the model doesn't translate.

### CLI / settings wiring

- `cli.py` reads `settings.raw["lsp"]`, instantiates `LSPClient`,
  calls `.configure()` (no spawn yet -- lazy on first tool call).
- New `--no-lsp` flag + `OPEN_CODE_NO_LSP` env var to disable.
- Settings shape:

```json
{
  "lsp": {
    "enabled": true,
    "servers": {
      "python": {
        "command": "pyright-langserver",
        "args": ["--stdio"],
        "file_patterns": ["*.py", "*.pyi"]
      }
    }
  }
}
```

## Bugs caught during live testing

Three things broke between mock-probe-green and live-pyright-green:

### 1. Pyright is slow on cold start (15-20s acceptable)

The mock LSP server replies in microseconds. Pyright's TypeScript-
based language server JIT-compiles + scans the project on first
didOpen. My initial `DIAGNOSTICS_WAIT_SECS=5` was too tight.

Fix: bumped to 20. Most calls return in 3-5s; outliers
(very-cold-start, large project scan) need the full budget.

### 2. Pyright publishes diagnostics in TWO phases

After `didOpen`, pyright emits a `publishDiagnostics` with an
EMPTY list ("I'm processing"), THEN the real list 1-3s later. My
initial code grabbed the first publish and returned `[]`.

Fix: replaced "wait for first notification" with a poll-until-
quiescent loop. Snapshot the diagnostics list every 250ms; once
the same non-empty list has been stable for >=1s, return. Times
out after `DIAGNOSTICS_WAIT_SECS` regardless.

### 3. URI encoding mismatch (the head-scratcher)

My `path_to_uri` produces `file:///C:/Users/kleiy/...`. Pyright
echoes back `file:///c%3A/Users/kleiy/...` (lowercase drive +
percent-escaped colon, per RFC 3986). I was using the URI as a
dict key for diagnostics, so my lookup key never matched what
pyright wrote.

Diagnostic that surfaced this: a 20-line debug script that printed
both forms side-by-side.

Fix: introduced `_canonicalize_uri(uri)` -- round-trips through
`Path(uri_to_path(uri)).resolve().as_uri()`. Both forms collapse
to the same canonical Windows-with-uppercase-drive shape. Applied
at the dispatcher's publishDiagnostics store site. Lookups use
my `path_to_uri` directly which already produces the canonical form.

## Honest trade-offs

- **Single-language config in the smoke** -- pyright only. The
  README documents rust-analyzer / gopls / typescript-language-
  server snippets but those aren't wire-verified. Each LSP server
  has its own quirks (especially around cold-start latency + how
  it handles workspace/configuration requests); odds are good the
  others have at least one wrinkle the way pyright did.

- **No incremental sync** -- I use full-text didChange (entire file
  re-sent on every update). LSP supports incremental edits but the
  spec is finicky and we only need this for read-only intelligence
  queries, not for the model to edit files live. Full-text is fine.

- **No persistent file watcher** -- if you edit a file outside open-
  code while a session is running, the LSP server doesn't know. The
  agent's next tool call re-sends with didChange, so this works
  out as a side effect. But there's no `didChangeWatchedFiles`
  notification handling.

- **One server per language** -- you can't run two pyright instances
  with different configs simultaneously (e.g. two Python projects
  side-by-side under one rootUri). That's a rarely-needed feature;
  punt until requested.

- **publishDiagnostics-during-call race is benign but real.** If
  pyright publishes new diagnostics WHILE a hover/def/refs call is
  in flight on a different position in the same file, the call's
  result reflects the freshly-updated state, not the state at
  call-time. This is actually preferable for an agent's correctness
  story, but it's worth noting that the tools aren't transactional
  snapshots.

- **Workspace/configuration requests get null responses.** Some
  servers send `workspace/configuration` to the client to ask for
  per-language settings. I reply `result: null` (defaults), which
  pyright handles gracefully. More finicky servers (rust-analyzer's
  initialization options) may need a real response someday.

## What this unlocks

The agent can now ask, BEFORE editing a file:

  "Where else is `_canonicalize_uri` used?" -> `lsp_references`
  "What's the type of this thing?" -> `lsp_hover`
  "Anything currently broken in this module?" -> `lsp_diagnostics`

That's the "agent makes fewer wrong-file edits" win the inventory
called the biggest single Tier 3 lever.
